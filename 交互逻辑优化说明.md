# 微信小程序、Python 后端与 CSI 硬件交互优化说明

本文以根目录 `hardware代码交互逻辑说明.md` 为唯一硬件协议基线。

## 1. 当前有效工程

- 微信小程序：`csi_miniprogram--/miniprogram`
- Python/数据库/MQTT 后端：`ESP32-MQTT-Backend-main`
- `csi_miniprogram--/backend` 是早期 FastAPI 演示实现，不具备当前
  微信登录、数据库设备归属和 v1 Topic 状态机，不应与正式 Flask 后端同时运行。

## 2. 登录与设备归属

```text
小程序冷启动
→ 优先使用已有 access_token 请求 Flask
→ Token 不可用或首次启动时调用 wx.login
→ POST /api/v1/auth/wechat-login
→ 后端按 openid 查询或创建 users
→ 管理员在 devices.owner_user_id 绑定设备
→ 小程序只读取当前用户的设备
```

首次启动现在允许后端创建用户；再次启动优先恢复 Token，Token 失效后重新
执行 code2Session，同一 openid 不会重复建档。

## 3. MQTT 状态映射

| 硬件上报 | 数据库/小程序结果 |
|---|---|
| `up/online {"status":"online"}` | `devices.state=online` |
| `up/online {"status":"offline"}` | `state=offline`，检测回到 idle |
| `up/status state=booting` | runtime=booting，检测 idle |
| `up/status state=idle` | runtime=idle；非控制等待状态下检测 idle |
| `up/status state=uploading` | runtime=uploading，检测 running |
| `up/status state=fault` 或 `up/fault` | `state=error`，记录 code/msg |
| `up/ack start ok=true` | starting → running |
| `up/ack stop ok=true` | stopping → idle |

HTTP 控制接口返回 202 只表示 QoS 1 消息已交给 MQTT 客户端。后端不会在
发布时直接假定硬件已经启停，最终状态由 ACK 确认。

下行消息固定为：

```json
{"cmd":"control","action":"start","session":"sess-..."}
```

停止时复用同一个 session。后端幂等 Key 不再错误占用硬件 `cmd` 字段。

## 4. 故障闭环

```text
硬件 up/fault
→ 后端 state=error、runtime=fault
→ WebSocket 立即通知当前用户
→ 后端自动发布同 session 的 stop
→ 硬件 up/ack(action=stop)
→ detection_state=idle，但保留 error 和 fault 信息
→ 设备重新发布 online 后清除旧故障
```

同一故障事件在 Python 侧也做 5 秒限流，自动 stop 对同一 session 去重。

## 5. CSI 解包与网络质量

后端处理 `up/csi` 的顺序为：

```text
校验 active session
→ 校验 fmt=csib64-v2-1s
→ Base64 严格解码及 bytes 长度校验
→ 解析 8 字节 batch header
→ 逐帧解析 16 字节 frame header + raw CSI
→ 交叉校验 frames/batch/seq0/seq1/ts0/ts1/invalid
→ 滚动计算丢帧、无效帧和 1 秒批次稳定性
→ 将已解码逐帧 raw_csi 送入跌倒算法接口
```

原始 CSI 只保留在有限内存窗口中，不写数据库。当前跌倒算法仍保持安全占位
实现，返回 0；后续只需替换 `fall_detect_service.py::predict_fall()`。

## 6. 联调验收顺序

1. 启动 Mosquitto 和 Flask，确认数据库已有管理员绑定的 `device_name`。
2. C 板上线后检查 online、wifi、status 是否更新设备快照。
3. 小程序点击开始，确认收到 `down/control` 且 payload 的 `cmd=control`。
4. 检查 start ACK 后 UI 进入 running，而不是在 HTTP 202 时提前进入。
5. 连续接收两包以上 CSI，确认网络质量从 unknown 进入 good/fair/poor。
6. 断开 B→C UART，确认 UART_TIMEOUT、自动 stop 和小程序异常提示。
7. 恢复并重启 C 板，确认 online 清除旧故障；再验证主动 stop ACK。
