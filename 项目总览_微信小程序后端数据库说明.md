# 项目总览：微信小程序 UI、Python 后端、数据库与 MQTT 业务逻辑

本文用于让 ChatGPT 或新的开发者快速理解当前项目结构、核心业务流程、前后端交互、数据库字段含义以及硬件 MQTT 协议关系。

## 1. 项目定位

这是一个基于微信小程序、Flask 后端、SQLite/SQLAlchemy 数据库和 Mosquitto MQTT 的 CSI 跌倒检测系统。

整体链路：

```text
微信小程序 UI
  -> Flask /api/v1 用户接口
  -> SQLAlchemy 数据库
  -> Flask DeviceCoordinator
  -> MQTT Broker / Mosquitto
  -> ESP32 C 板 / 硬件设备
  -> MQTT 上报 online/status/csi/ack/fault
  -> Flask 更新数据库并通过 WebSocket 推送给小程序
```

项目当前包含两个主要目录：

```text
csi_miniprogram--/
  微信小程序前端，TypeScript + WXML + LESS

ESP32-MQTT-Backend-main/
  Python Flask 后端，SQLAlchemy 数据库，MQTT 接入，WebSocket 推送
```

根目录还包含业务和硬件说明文档：

```text
hardware代码交互逻辑说明.md
交互逻辑优化说明.md
硬件代码检测指南.md
QUICK_START.md
```

### Python 后端

后端入口：

```text

```

当前入口大致逻辑：

```python
from dotenv import load_dotenv
from app import create_app

load_dotenv()
app = create_app()

if __name__ == "__main__":
    app.run(debug=True)
```

应用工厂：

```text
ESP32-MQTT-Backend-main/app/__init__.py
```

它负责：

```text
1. 创建 Flask app
2. 加载配置
3. 初始化 SQLAlchemy 和 WebSocket sock
4. 注册后台管理 API /api
5. 注册小程序 API /api/v1
6. 注册 WebSocket /ws/v1/events
7. 创建 DeviceCoordinator
8. 根据 MQTT_AUTOSTART_DEVICES 决定是否启动全部设备 MQTT 订阅
9. 启动离线扫描线程
```

重要提醒：如果 Flask debug reloader 导致 MQTT client 重复连接，可以将 `app.run(debug=True)` 改成：

```python
app.run(debug=True, use_reloader=False)
```

数据库模型定义在：

主要字段：

```text

```

### 4.2 users 表

模型：`User`

用途：微信小程序用户身份、手机号、账号状态、设备归属。

主要字段：

```text
id
wx_openid             微信 openid，唯一
wx_unionid            微信 unionid，可为空
wx_session_key_enc    当前项目不保存明文 session_key，通常为空
nickname
avatar_url
phone
role                  user/admin
status                active/disabled
last_login_at
created_at
updated_at
```

关系：

```text
User 1 -> N Device
User 1 -> N FallEvent
```

### 4.3 devices 表

模型：`Device`

用途：设备绑定关系、展示信息、在线状态、检测状态、网络质量、故障信息。

主要字段：

```text
id
device_name           设备唯一编号，也是 MQTT topic 中的设备名
display_name          小程序展示名称
owner_user_id         绑定用户 id
location              安装位置
remark                管理员备注
enabled               管理员是否启用设备

state                 online/offline/error
runtime_state         booting/idle/uploading/fault
detection_state       idle/starting/running/stopping
current_session       当前检测 session
network_quality       good/fair/poor/unknown

fault_code
fault_message
last_seen_at
last_online_at
last_status_at
last_csi_at
created_at
updated_at
```

重要状态含义：

```text
state:
  online   设备在线
  offline  设备离线
  error    设备故障/异常

runtime_state:
  booting    硬件启动中
  idle       硬件空闲
  uploading  硬件正在上传 CSI
  fault      硬件进入故障状态

detection_state:
  idle      小程序侧可启动检测
  starting  后端已下发 start，等待硬件 ACK 或 status/csi
  running   检测运行中
  stopping  后端已下发 stop，等待硬件停止

network_quality:
  good     CSI 网络质量良好
  fair     一般
  poor     较差
  unknown  暂无数据
```

`device_name` 会生成 MQTT topic：

```text
csi/v1/devices/{device_name}/up/csi
csi/v1/devices/{device_name}/down/control
```

### 4.4 fall_events 表

模型：`FallEvent`

用途：保存跌倒检测事件。

主要字段：

```text
id
user_id
device_id
device_name
session
result                 当前 result=1 表示检测到跌倒
network_quality
occurred_at
status                 pending/confirmed/ignored
notified
notified_at
handled_at
remark
created_at
updated_at
```

小程序里用户确认安全后，会将事件状态改成 `confirmed`。

## 5. 小程序前端业务结构



全局数据：

```ts

  user

  fallEvents

  bootstrapReady
  bootstrapError
  isNewUser
```

### 5.5 pages/device-detail：设备检测详情页

文件：

```text
csi_miniprogram--/miniprogram/pages/device-detail/index.ts
```

功能：

```text
1. GET /api/v1/devices/{device_name} 获取设备详情
2. 展示设备状态、网络质量、最近通信时间
3. 根据状态显示呼吸灯：
   - normal 绿色：正常/可检测/运行中
   - fall 红色：存在 pending 跌倒事件
   - error 黄色：设备异常
   - offline 灰色静止：设备离线
4. 点击开始检测：
   - POST /api/v1/devices/{device_name}/control { action: "start" }

5. 点击停止检测：
   - POST /api/v1/devices/{device_name}/control { action: "stop" }
   - 前端先进入 stopping 状态
   - 轮询/等待 WebSocket 更新到 idle
6. 若有 pending 跌倒事件，支持进入告警详情或确认安全
```

按钮禁用条件：

```text
controlLoading
device.enabled=false
device.state != online
device.fault.code 存在
device.detection_state 为 starting/stopping
```

## 6. 后端 API

### 6.1 小程序 API：/api/v1

文件：

```text
ESP32-MQTT-Backend-main/app/miniapp_api.py
```

认证方式：

```text
Authorization: Bearer <access_token>
```

token 由：

```text
ESP32-MQTT-Backend-main/app/services/token_service.py
```

签发和校验。

#### 登录

```http
POST /api/v1/auth/wechat-login
```

请求：

```json
{
  "code": "wx.login 返回的 code",
  "create_if_missing": true
}
```

后端逻辑：

```text
1. code 不能为空
2. 调用微信 code2session 换 openid/unionid
3. users 表按 wx_openid 查用户
4. 不存在则创建 User
5. 更新 last_login_at
6. 签发 access_token
```

返回：

```json
{
  "access_token": "...",
  "expires_in": 7200,
  "user": {
    "id": 1,
    "nickname": null,
    "avatar_url": null,
    "is_new_user": true
  }
}
```

#### 当前用户

```http
GET /api/v1/me
PATCH /api/v1/me/profile
POST /api/v1/me/phone
```

`/me/phone` 用于微信手机号授权，将手机号写入 `users.phone`。

#### 当前用户设备

```http
GET /api/v1/devices
```

逻辑：

```text
1. 查询 devices.owner_user_id == 当前用户 id 的设备
2. 调用 DeviceCoordinator.ensure_for_user(user_id)
3. 为该用户 enabled=true 的设备建立 MQTT client
4. 返回设备摘要
```

```http
GET /api/v1/devices/{device_name}
```

逻辑：

```text
1. 校验设备存在
2. 校验设备属于当前用户
3. 若 device.enabled=true，确保 MQTT client 已建立
4. 返回设备详情
```

```http
PATCH /api/v1/devices/{device_name}
```

用户可编辑字段：

```text
display_name  必填时不能为空，最长 64
location      可为空，最长 128
```

安全边界：

```text
用户不能改 device_name
用户不能改 owner_user_id
用户不能改 enabled
用户不能改 state/runtime_state/detection_state/current_session
用户不能改 MQTT topic
```

#### 设备控制

```http
POST /api/v1/devices/{device_name}/control
```

请求：

```json
{ "action": "start" }
```

或：

```json
{ "action": "stop" }
```

前端会带：

```text
Idempotency-Key: <随机请求 id>
```

后端用它做 5 分钟幂等缓存，避免重复点击导致重复控制。

#### 跌倒事件

```http
GET /api/v1/fall-events?limit=20
PATCH /api/v1/fall-events/{event_id}
```

`PATCH` 只允许当前用户处理自己的事件，状态只能改成：

```text
confirmed
ignored
```

## 7. MQTT 与硬件交互

### 7.1 Topic 规范

定义位置：

```text
ESP32-MQTT-Backend-main/app/mqtt/topics.py
```

topic 前缀：

```text
csi/v1/devices
```

设备上行 topic：

```text
csi/v1/devices/{device_name}/up/online
csi/v1/devices/{device_name}/up/wifi
csi/v1/devices/{device_name}/up/status
csi/v1/devices/{device_name}/up/csi
csi/v1/devices/{device_name}/up/ack
csi/v1/devices/{device_name}/up/fault
```

后端下行 topic：

```text
csi/v1/devices/{device_name}/down/control
```

QoS：

```text
online  QoS 1
wifi    QoS 1
status  QoS 1
csi     QoS 0
ack     QoS 1
fault   QoS 1
control QoS 1
```

### 7.2 MQTT client 设计

文件：

```text
ESP32-MQTT-Backend-main/app/mqtt/client.py
ESP32-MQTT-Backend-main/app/services/mqtt_service.py
```

当前设计：

```text
1. 一个 database device_name 对应一个 Python MQTT client
2. client_id 格式：python_{device_name}_{instance:03d}
3. 默认 instance=1，因此示例：python_esp01_001
4. 连接成功后订阅该设备 6 个 up topic
5. 控制时向 down/control 发布 JSON
```

`MQTT_AUTOSTART_DEVICES=False` 时：

```text
后端启动不会立刻连接所有设备的 MQTT。
用户打开设备列表/详情或发送控制命令时，才 ensure_device。
```

### 7.3 control 下行格式

发布位置：

```text
DeviceMqttClient.publish_control()
```

payload：

```json
{
  "cmd": "control",
  "action": "start",
  "session": "sess-..."
}
```

停止：

```json
{
  "cmd": "control",
  "action": "stop",
  "session": "sess-..."
}
```

硬件需要订阅：

```text
csi/v1/devices/{device_name}/down/control
```

### 7.4 硬件上报 online

后端处理函数：

```text
DeviceCoordinator._handle_online()
```

可接受字段：

```json
{ "status": "online" }
```

或：

```json
{ "online": true }
```

上线时数据库更新：

```text
devices.state = online
devices.last_seen_at = now
devices.last_online_at = now
清空 fault_code/fault_message
```

离线时数据库更新：

```text
devices.state = offline
devices.runtime_state = idle
devices.detection_state = idle
devices.current_session = null
devices.network_quality = unknown
```

并推送 WebSocket：

```text
device.state.changed
```

### 7.5 硬件上报 status

后端处理函数：

```text
DeviceCoordinator._handle_status()
```

硬件状态必须是：

```text
booting
idle
uploading
fault
```

示例：

```json
{
  "state": "uploading",
  "session": "sess-..."
}
```

`uploading` 时：

```text
runtime_state = uploading
state = online
detection_state 通常进入 running
last_seen_at/last_status_at 更新
```

`idle` 且当前为 `stopping` 时：

```text
detection_state = idle
current_session = null
network_quality = unknown
```

`fault` 时进入故障逻辑。

并推送：

```text
device.runtime.changed
```

### 7.6 硬件上报 ack

后端处理函数：

```text
DeviceCoordinator._handle_ack()
```

格式：

```json
{
  "cmd": "control",
  "action": "start",
  "ok": true,
  "session": "sess-...",
  "state": "uploading"
}
```

或停止：

```json
{
  "cmd": "control",
  "action": "stop",
  "ok": true,
  "session": "sess-...",
  "state": "idle"
}
```

start ack 成功：

```text
runtime_state = uploading
detection_state = running
state = online
```

stop ack 成功：

```text
runtime_state = idle
detection_state = idle
current_session = null
network_quality = unknown
```

失败 ack：

```text
start 失败 -> detection_state 回 idle
stop 失败  -> detection_state 回 running
```

推送：

```text
device.runtime.changed
```

其中 data 会包含：

```text
control_ok
action
err
message
```

### 7.7 硬件上报 fault

后端处理函数：

```text
DeviceCoordinator._handle_fault()
DeviceCoordinator._record_fault()
```

格式：

```json
{
  "code": "NO_CSI_FRAME",
  "msg": "no csi frame from b board"
}
```

或：

```json
{
  "fault_code": "NO_CSI_FRAME",
  "fault_message": "no csi frame from b board"
}
```

数据库更新：

```text
state = error
runtime_state = fault
fault_code = code
fault_message = message
last_seen_at = now
```

如果当时正在 starting/running/stopping 且有 session：

```text
detection_state = stopping
network_quality = unknown
后端自动发布 control stop
```

推送：

```text
device.fault
```

### 7.8 CSI 数据上报

后端处理函数：

```text
DeviceCoordinator._handle_csi()
```

解析文件：

```text
ESP32-MQTT-Backend-main/app/services/csi_payload_service.py
```

当前支持格式：

```text
fmt = csib64-v2
```

MQTT JSON 示例字段：

```json
{
  "session": "sess-...",
  "batch": 1,
  "frames": 30,
  "seq0": 1,
  "seq1": 30,
  "ts0": 12345678,
  "ts1": 13345678,
  "fmt": "csib64-v2",
  "bytes": 7528,
  "data": "Base64...",
  "ts": 123
}
```

二进制结构：

```text
BatchHeader 28 bytes:
  <4sBBBBIqq
  magic="CSIB"
  version=0x01
  frame_count
  sample_hz
  reserved=0
  batch_seq
  start_ts_us
  end_ts_us

FrameHeader 16 bytes:
  <IqbBH
  seq
  timestamp_us
  rssi
  first_word_invalid
  csi_len

后面跟 csi_len 字节 raw CSI。
```

约束：

```text
最大 batch frames = 80
最大单帧 CSI bytes = 384
JSON 不再要求 rssi/invalid，后端从 frame header 计算 average_rssi 和 invalid_count
```

CSI 处理逻辑：

```text
1. 必须 device.state=online
2. detection_state 必须 starting/running
3. device.current_session 必须存在
4. payload.session 必须等于 current_session
5. 解析 csib64-v2
6. 更新 last_seen_at/last_csi_at/network_quality
7. 将 batch 放入滑动窗口
8. 达到 CSI_WINDOW_SIZE 后调用 predict_fall
9. 如果检测到跌倒，写入 fall_events 并推送 detection.fall-result
```

## 8. 核心业务流程

### 8.1 用户首次登录

```text
小程序 app.ts
  -> wx.login 获取 code
  -> POST /api/v1/auth/wechat-login
  -> 后端 exchange_wechat_code
  -> users 表按 wx_openid 查询
  -> 不存在则创建 User
  -> 更新 last_login_at
  -> 签发 access_token
  -> 小程序保存 token
  -> bootstrapBusinessData 加载 me/devices/fall-events
  -> WebSocket connect
```

首次登录后，数据库至少会有：

```text
users.wx_openid
users.last_login_at
users.status=active
```

手机号不是强制，但我的页面会提示授权。授权后：

```text
POST /api/v1/me/phone
-> users.phone 更新
```

### 8.2 管理员绑定设备

```text
管理员后台 /api/devices
  -> POST /api/devices
  -> 提交 device_uid/name/owner_id/status/location/remark
  -> 写入 devices 表
  -> devices.owner_user_id 指向 users.id
```

用户下一次打开小程序：

```text
GET /api/v1/devices
-> 查询 owner_user_id == 当前用户 id
-> 返回设备列表
```

### 8.3 用户查看设备状态

```text
小程序首页
  -> GET /api/v1/devices
  -> 后端 ensure_for_user
  -> 为用户 enabled 设备建立 MQTT client
  -> MQTT client 订阅 online/wifi/status/csi/ack/fault
  -> 返回 devices.to_summary_dict()
```

设备状态主要由硬件上报驱动：

```text
online topic -> state online/offline
status topic -> runtime_state/detection_state
fault topic  -> state error
csi topic    -> last_csi_at/network_quality/fall_event
ack topic    -> 控制命令最终状态
```

### 8.4 用户开始检测

```text
小程序设备详情页点击开始检测
  -> POST /api/v1/devices/{device_name}/control { action: "start" }
  -> DeviceCoordinator.control_device()
  -> 校验设备属于用户
  -> 校验 enabled=true
  -> 校验 state=online
  -> 校验无 fault
  -> 校验 last_status_at 未超时
  -> 校验 detection_state=idle
  -> 生成 session
  -> MQTT publish down/control start
  -> devices.current_session=session
  -> devices.detection_state=starting
```

硬件收到 start 后应：

```text
1. 启动 CSI 上传
2. 发布 ack ok=true 或 status uploading
3. 定期发布 status/csi
```

后端收到 ack/status/csi 后：

```text
ack start ok=true       -> detection_state=running
status uploading        -> detection_state=running
csi 当前 session 有效    -> 更新 network_quality，进入算法窗口
WebSocket 推送给小程序
```

小程序表现：

```text
starting -> 正在启动
running  -> 检测运行中，绿色呼吸灯
```

### 8.5 用户停止检测

```text
小程序设备详情页点击停止检测
  -> POST /api/v1/devices/{device_name}/control { action: "stop" }
  -> 后端要求 detection_state in starting/running
  -> 要求 current_session 存在
  -> MQTT publish down/control stop
  -> devices.detection_state=stopping
```

硬件收到 stop 后应：

```text
1. 停止 CSI 上传
2. 发布 ack ok=true state=idle
3. 或发布 status idle
```

后端收到后：

```text
ack stop ok=true 或 status idle
  -> runtime_state=idle
  -> detection_state=idle
  -> current_session=null
  -> network_quality=unknown
  -> 清理该 session 的 CSI 窗口
```

### 8.6 CSI 正常上传和网络质量

```text
硬件持续发布 up/csi
  -> 后端解析 csib64-v2
  -> CsiQualityTracker 计算质量
  -> devices.network_quality = good/fair/poor/unknown
  -> last_csi_at 更新
  -> 如果质量变化，推送 detection.network-quality
```

如果运行中长时间没有 CSI：

```text
DeviceCoordinator.scan_offline_devices()
  -> 如果 detection_state=running
  -> last_csi_at 超过 CSI_TIMEOUT_SECONDS
  -> network_quality 改成 poor
  -> 推送 detection.network-quality
```

### 8.7 跌倒检测

当前跌倒算法入口：

```text
ESP32-MQTT-Backend-main/app/services/fall_detect_service.py
```

调用位置：

```text
DeviceCoordinator._handle_csi()
```

流程：

```text
1. 每个 device_name + session 维护一个 CSI 滑动窗口
2. 窗口长度达到 CSI_WINDOW_SIZE
3. 调用 predict_fall(device_name, session, window)
4. 如果返回 1：
   - 创建 fall_events
   - status=pending
   - notified=true
   - 推送 detection.fall-result
5. 同一个 session 只发一次跌倒事件
```

小程序收到 `detection.fall-result`：

```text
1. 首页或详情页跳转到 fall-alert
2. 用户可确认安全
3. PATCH /api/v1/fall-events/{id} status=confirmed
```

### 8.8 设备故障

硬件发布 fault：

```text
csi/v1/devices/{device_name}/up/fault
```

后端：

```text
state=error
runtime_state=fault
fault_code/fault_message 写入数据库
如果检测正在运行，自动下发 stop
推送 device.fault
```

小程序：

```text
设备详情页显示黄色呼吸灯
设备控制按钮禁用
展示故障信息
```

### 8.9 设备离线

离线来源有两种：

```text
1. 硬件主动发布 online=false/offline
2. 后端离线扫描发现 last_seen_at 超过 OFFLINE_TIMEOUT_SECONDS
```

后端更新：

```text
state=offline
runtime_state=idle
detection_state=idle
current_session=null
network_quality=unknown
```

小程序：

```text
显示灰色静止灯
控制按钮禁用
```

## 11. 快速理解当前设计的几个关键点

1. `devices` 表是设备业务状态的权威快照，小程序展示基本都来自这里。
2. `users` 表由微信登录创建，设备绑定由管理员写入 `devices.owner_user_id`。
3. 小程序用户不能自己绑定新设备，只能查看管理员绑定给自己的设备。
4. 小程序用户现在可以修改自己的设备 `display_name` 和 `location`。
5. 后端默认不一定启动时连接所有 MQTT 设备，取决于 `MQTT_AUTOSTART_DEVICES`。
6. 用户打开设备列表/详情时，会触发后端为其设备建立 MQTT 订阅。
7. 控制设备以硬件 ACK/status 为最终准，HTTP 202 只表示命令已发布。
8. `state` 表示设备在线/离线/异常，`runtime_state` 表示硬件运行状态，`detection_state` 表示小程序检测流程状态。
9. CSI 数据只有在当前 session 匹配、设备 online、检测 starting/running 时才会被处理。
10. 跌倒事件写入 `fall_events` 后，通过 WebSocket 推送给小程序。
