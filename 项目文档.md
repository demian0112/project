# CSI 跌倒检测项目文档

> 文档基准：2026-07-10 当前工作区代码。本文以实际运行入口和当前可见 UI 为准；仅有接口或保留代码、但尚未接入界面的能力会单独标注。

## 1. 项目概览

本项目由两个主要子项目组成：

- `csi_miniprogram--`：原生微信小程序前端，使用 TypeScript、WXML、Less 开发。
- `ESP32-MQTT-Backend-main`：Flask 后端，负责小程序接口、后台管理页面、管理员 CSI 实时热力图、SQLite 数据库、WebSocket 实时推送，以及与 ESP32/采集设备之间的 MQTT 通信。

整体业务目标是：用户在小程序中查看家庭设备状态、启动或停止 CSI 检测；硬件通过 MQTT 上传在线状态、运行状态、CSI 数据、控制回执和故障；后端维护设备权威状态快照，检测跌倒结果并推送到小程序。

核心数据链路：

```text
微信小程序
  -> /api/v1 REST 接口
  -> Flask 后端 / SQLite
  -> DeviceCoordinator
  -> MQTT Broker
  -> ESP32/CSI 采集设备

ESP32/CSI 采集设备
  -> MQTT 上行主题
  -> Flask 后端解析与状态更新
  -> SQLite 状态快照 / WebSocket 推送
  -> 小程序实时刷新和告警跳转
  -> 内存 CSI 实时缓冲区
  -> 管理员设备页每秒拉取并绘制热力图
```

当前正式业务链路只使用 `ESP32-MQTT-Backend-main`。`csi_miniprogram--/fall_detector` 位于小程序构建目录之外，仍依赖已经删除的旧 `backend/config.py`，且使用旧 MQTT topic/payload 协议，因此不属于当前可运行链路。

## 2. 前端设计

### 2.1 小程序结构

小程序为原生项目，`miniprogramRoot` 是 `miniprogram/`，由微信开发者工具编译 TypeScript 和 Less。当前使用 `style: v2`、`glass-easel`、按需组件加载，项目基础库版本为 2.32.3；TypeScript 目标为 ES2017，并开启严格类型和未使用变量检查。npm 脚本目前只有 `npm run typecheck`。

小程序页面注册在 `miniprogram/app.json`：

- `pages/index/index`：监测首页。
- `pages/device-detail/index`：设备详情与启动/停止控制。
- `pages/device-management/index`：设备资料管理。
- `pages/fall-alert/index`：跌倒告警详情。
- `pages/mine/mine`：个人中心、登录状态、设备数量和设备管理入口。

全局配置 `navigationStyle: custom`，各页面自行实现顶部区域；实际复用的是自定义 tabBar，底部主入口只有：

- 监测：首页。
- 我的：个人中心。

组件现状：

- `custom-tab-bar`：当前实际使用的自定义底部导航。
- `components/gauge-ring`：保留的 Canvas 2D 环形仪表盘组件，当前页面未引用。
- `components/navigation-bar`：保留的导航栏组件，当前页面未引用。

### 2.2 环境与接口配置

配置位于 `miniprogram/config/env.ts`。

开发环境默认：

```text
apiBaseUrl = http://127.0.0.1:5000
wsBaseUrl  = ws://127.0.0.1:5000/ws/v1/events
```

小程序 token 存储 key：

```text
anshou_csi_access_token
```

订阅消息模板 ID 在同一文件中配置，当前包含：

- 跌倒提醒模板：`fallAlert`
- 设备故障模板：`deviceFault`

### 2.3 启动与登录流程

入口逻辑在 `miniprogram/app.ts`。

小程序启动时执行：

1. 清空业务缓存状态。
2. 从本地存储读取 token。
3. 如果 token 存在，设置全局登录态并拉取业务数据。
4. 如果 token 不存在，或 401 等认证错误已经清除认证态，调用 `wx.login`；普通网络错误只记录 `bootstrapError`，不会直接重新登录。
5. 将微信 `code` 发送到后端 `/api/v1/auth/wechat-login`。
6. 后端返回 access token 后写入本地存储。
7. 并发拉取用户、设备和最近 20 条跌倒事件。
8. 建立 WebSocket 实时连接。

全局数据主要包括：

- `token`
- `user`
- `devices`
- `fallEvents`
- `authReady`
- `bootstrapReady`
- `bootstrapError`

认证等待逻辑在 `services/auth.ts`。接口请求除登录接口外，都会先等待 `authReady`，避免页面在登录未完成时抢先请求业务接口。

### 2.4 API 契约层

前端所有 REST 调用集中在 `miniprogram/utils/api.ts`。

请求封装负责：

- 拼接 `runtimeConfig.apiBaseUrl`。
- 添加 `Authorization: Bearer <token>`。
- 统一超时时间。
- 处理 401：清除 token、重置认证态、提示重新登录。
- 将后端错误转换为 `ApiError`。

主要接口函数：

- `loginWithWechat(code, createIfMissing)`
- `getCurrentUser()`
- `updateCurrentUserProfile(profile)`：已有封装，当前没有可见 UI 调用。
- `updateCurrentUserPhone(code)`：已有封装和事件处理代码，但当前手机号卡片已隐藏。
- `getDevices()`
- `getDeviceDetail(deviceName)`
- `updateDeviceInfo(deviceName, profile)`
- `controlDevice(deviceName, action)`
- `resetDeviceFault(deviceName)`：已有封装，当前设备详情页没有复位入口。
- `getFallEvents(limit)`
- `updateFallEvent(id, status)`
- `registerWechatSubscription(payload)`
- `getWechatSubscriptionStatus()`

主要请求/响应结构：

| 操作 | 请求核心字段 | 响应核心字段 |
| --- | --- | --- |
| 微信登录 | `code`、`create_if_missing` | `access_token`、`expires_in`、`user{id,nickname,avatar_url,is_new_user}` |
| 用户资料 | 更新时提交 `nickname?`、`avatar_url?`；手机号提交一次性 `code` | `id`、`nickname`、`avatar_url`、`phone`、`status`、`last_login_at` |
| 订阅状态 | POST 提交 `scene`、`template_id`、`status` | GET 返回 `ok`、`enabled`、默认场景快照及 `items[]` 两场景快照；POST 返回保存后的场景、模板、状态和 `remaining_count` |
| 设备列表 | 无 | `{items: DeviceSummary[]}` |
| 设备详情/资料 | 更新时提交 `display_name?`、`location?` | 基本字段及嵌套 `runtime`、`detection`、`fault`；前端再规范化为 `DeviceDetail` |
| 设备控制 | `action: start|stop`，并发送 `Idempotency-Key` | `accepted`、`device_name`、`action`、`control_state`、`session`、`message` |
| 故障复位 | 空 JSON 和 `Idempotency-Key` | 与控制响应相同；当前 UI 未调用 |
| 跌倒事件 | 列表 query `limit`；更新提交 `status` | 列表 `{items: FallEvent[]}`；更新 `{success,item}`，前端只声明使用 `success` |

设备摘要核心字段为 `device_name`、`display_name`、`location`、`state`、`detection_state`、`network_quality`、`last_seen_at`、公开故障信息；详情另含 `remark`、`enabled`、`runtime{state,last_status_at}`、`detection{state,session,network_quality,last_csi_at}`、`fault{code,message,template_data}`。

后端通用错误体是 `{error, message}`，前端转换为 `ApiError`，附带 `code` 和 HTTP `statusCode`。

### 2.5 实时通信设计

WebSocket 客户端位于 `miniprogram/services/realtime.ts`。

连接方式：

```text
ws://127.0.0.1:5000/ws/v1/events?token=<access_token>
```

前端设计要点：

- token 通过 query 参数传给后端鉴权。
- 连接成功后通知页面重新加载数据。
- 消息统一解析为 `RealtimeEvent`。
- 连接异常时自动重连，重连间隔指数退避，最大 30 秒。
- 只有 `wsBaseUrl` 为空时，首页和设备详情页才启用 5 秒轮询；地址已配置但临时断线时依赖 WebSocket 自动重连，不会同时启用轮询。

实时事件类型定义在 `models/domain.ts`，包括：

- `connection.ready`
- `device.state.changed`
- `device.runtime.changed`
- `device.fault`
- `detection.network-quality`
- `detection.fall-result`

### 2.6 页面交互逻辑

#### 2.6.1 监测首页

代码位于 `pages/index/index.ts`。

首页展示：

- 全部设备数量。
- 在线设备数量。
- 检测中设备数量。
- 最新一条跌倒事件；只有该事件状态为 `pending` 时才显示告警。首页不会继续向后搜索更早的待处理事件。
- 设备列表、设备状态、检测状态、CSI 网络质量。

页面进入时：

1. 订阅 WebSocket 事件。
2. 加载设备列表。
3. 加载最近跌倒告警。
4. 未配置 `wsBaseUrl` 时开启 5 秒轮询。

收到实时事件时：

- `device.fault`：对 `NO_CSI_FRAME`、`NO_CSI_FRAME_TIMEOUT`、`UART_TIMEOUT` 三类 CSI 采集故障弹窗提示；设备详情页只展示异常信息，不提供复位操作。
- `detection.fall-result` 且 `fall_detected = true`：跳转到跌倒告警页。
- 其他状态事件：重新加载首页数据。

#### 2.6.2 设备详情页

代码位于 `pages/device-detail/index.ts`。

设备详情页是小程序中最重要的控制页面，负责：

- 展示设备在线/离线/异常。
- 展示检测状态。
- 展示 CSI 网络质量。
- 展示最近通信时间。
- 展示待处理跌倒告警。
- 可在详情页直接点击“我已确认安全”，把当前告警更新为 `confirmed`。
- 启动检测。
- 停止检测。

详情页只拉取用户最近 20 条跌倒事件，再查找本设备的 `pending` 记录，因此更早但仍待处理的告警可能不会出现在该页。

设备控制流程：

1. 用户点击“开始检测”或“停止检测”。
2. 前端先检查设备是否在线、是否启用、是否存在故障。
3. 开始检测前，读取微信订阅设置，并同时请求跌倒提醒和设备故障提醒两个模板；用户拒绝或授权调用失败不会阻止检测启动。
4. 前端先乐观地把页面置为 `starting` 或 `stopping`，设置 15 秒 pending 截止时间。
5. 前端调用 `/api/v1/devices/<device_name>/control`。
6. 后端返回 202，表示命令已经发布到 MQTT；前端随后启动短轮询，每 1.5 秒确认状态。
7. 如果控制请求失败，前端清除 pending 并重新加载设备状态。
8. 如果 WebSocket 收到运行状态变化，也会触发重新加载。
9. 设备最终进入 `running` 或 `idle` 后，前端解除 loading。

这里有一个重要设计：接口返回成功只代表后端已经发布控制命令，不代表硬件已经执行成功。最终状态以后端收到硬件 `ack`、`status` 或 `csi` 后更新为准。

当前页面在设备异常时只显示故障 banner，并禁用开始/停止按钮。后端 `/api/v1/devices/<device_name>/reset-fault`、前端 `resetDeviceFault()` 封装以及 MQTT `reset` 处理链路仍然存在，但当前 WXML 没有触发入口，不能把“故障复位”视为小程序当前可用功能。

另一个当前 UI 细节是：“通信与运行”区域的“运行状态”和“检测状态”都绑定了 `detectionText`，目前没有把 `runtime.state` 单独显示出来。

#### 2.6.3 跌倒告警页

代码位于 `pages/fall-alert/index.ts`。

页面通过 query 中的 `id` 加载跌倒事件。由于前端当前没有单条告警接口，页面会拉取最多 100 条事件，再在前端按 id 查找。

用户可以：

- 查看设备、位置、发生时间、网络质量。
- 点击“我已确认安全”，将事件状态更新为 `confirmed`。
- 点击“联系家人”只会出现占位说明，尚未接入联系人或拨号能力。
- 返回首页。

确认成功后页面会等待约 500 ms，再自动切换回“监测”首页。

#### 2.6.4 我的页面

代码位于 `pages/mine/mine.ts`。

主要能力：

- 展示用户资料。
- 如果未登录，显示登录入口。
- 查看设备数量。
- 进入设备管理页。

页面脚本仍会拉取订阅状态，也保留手机号授权和单独开启跌倒提醒的处理函数；但当前 `mine.wxml` 已注释手机号卡片，且没有订阅按钮或状态区域，因此用户界面上无法触发这两项能力。手机号接口仍按微信 `getPhoneNumber` 一次性 `code` -> `/api/v1/me/phone` -> 后端换取手机号的方式实现。

#### 2.6.5 设备管理页

代码位于 `pages/device-management/index.ts`。

普通用户可维护自己设备的：

- 显示名称。
- 安装位置。

该页面不支持绑定、解绑、启用或停用设备，这些操作由管理员后台负责。

前端会校验：

- 显示名不能为空。
- 显示名最多 64 字符。
- 安装位置最多 128 字符。

`create_app()` 负责：

后端主要模块：

### 3.2 配置设计

配置位于 `app/config.py`，支持 `.env` 环境变量覆盖。

主要配置包括：

- Flask 密钥和数据库地址。
- 管理员初始账号密码。
- 微信 AppID、Secret、code2Session、手机号接口。
- 微信订阅消息模板。
- token 过期时间。
- MQTT 是否启用。
- MQTT 是否自动启动设备客户端。
- 离线监控是否启用。
- 设备超时参数。
- CSI 质量检测参数。
- 故障事件限流参数。

主要环境变量分组：

- 应用与认证：`SECRET_KEY`、`DATABASE_URL`、`ADMIN_USERNAME`、`ADMIN_PASSWORD`、`TOKEN_EXPIRE_SECONDS`、`TOKEN_SALT`。其中 `.env.example` 未提供 `TOKEN_SALT`，生产环境需要手动新增。
- 微信身份与手机号：`WECHAT_APPID`、`WECHAT_SECRET`、`WECHAT_CODE2SESSION_URL`、`WECHAT_ACCESS_TOKEN_URL`、`WECHAT_PHONE_NUMBER_URL`。
- 微信订阅通知：`WECHAT_NOTIFY_ENABLED`、`WECHAT_FALL_ALERT_TEMPLATE_ID`、`WECHAT_DEVICE_FAULT_TEMPLATE_ID`、`WECHAT_SUBSCRIBE_SEND_URL`、`WECHAT_MINIPROGRAM_STATE`、`WECHAT_LANG`、`WECHAT_FALL_ALERT_PAGE`、`WECHAT_DEVICE_FAULT_PAGE`、`WECHAT_ACCESS_TOKEN_CACHE_SECONDS`。
- MQTT：`MQTT_ENABLED`、`MQTT_AUTOSTART_DEVICES`，以及 `MQTT_BROKER_HOST/PORT/USERNAME/PASSWORD`；后四项也兼容较短的 `MQTT_HOST/PORT/USERNAME/PASSWORD`，另有 `MQTT_KEEPALIVE`。
- 设备监控：`OFFLINE_MONITOR_ENABLED` 及下表中的 CSI/超时/限流参数。

关键开关：

```text
MQTT_ENABLED
MQTT_AUTOSTART_DEVICES
OFFLINE_MONITOR_ENABLED
WECHAT_NOTIFY_ENABLED
```

开关默认值为：`MQTT_ENABLED=false`、`MQTT_AUTOSTART_DEVICES=false`、`OFFLINE_MONITOR_ENABLED=true`、`WECHAT_NOTIFY_ENABLED=false`。因此默认会运行离线监控线程，但不会连接 MQTT 或发送微信服务通知。

当前代码默认值中的主要时间参数：

| 配置 | 默认值 | 当前用途 |
| --- | ---: | --- |
| `TOKEN_EXPIRE_SECONDS` | 7200 秒 | 小程序 access token 有效期 |
| `OFFLINE_TIMEOUT_SECONDS` | 15 秒 | 非检测状态离线判定 |
| `CSI_EXPECTED_INTERVAL_SECONDS` | 1.5 秒 | 计算正常数据间隔 |
| `CSI_SOFT_TIMEOUT_SECONDS` | 10 秒 | CSI 间隔过长时降级质量 |
| `CSI_RECOVERY_GRACE_SECONDS` | 20 秒 | 保持运行并等待恢复 |
| `CSI_HARD_TIMEOUT_SECONDS` | 35 秒 | 运行中无 CSI 时进入故障流程 |
| `START_GRACE_SECONDS` | 10 秒 | 启动阶段宽限 |
| `CSI_PARSE_ERROR_LIMIT` | 3 | 连续解析异常达到阈值后将质量置为 poor |
| `CSI_WINDOW_SIZE` | 5 帧 | 触发跌倒 predictor 的最小窗口帧数 |
| `FAULT_EVENT_LIMIT_SECONDS` | 5 秒 | 同类故障事件推送限流 |

`STATUS_TIMEOUT_SECONDS`、`CSI_TIMEOUT_SECONDS`、`RUNNING_IGNORE_STATUS_TIMEOUT` 当前会被加载到配置，但 `DeviceCoordinator` 没有读取它们；实际超时行为以上表中被使用的参数为准。

### 3.3 数据模型

核心模型位于 `app/models.py`，当前共映射六张表：`admin`、`users`、`devices`、`fall_events`、`wx_subscriptions`、`wx_notify_logs`。

#### Admin

后台管理员：

- `id`
- `username`
- `password_hash`
- `last_login_at`
- `created_at`

用于访问 `/admin` 管理控制台。

#### User

微信小程序用户：

- `id`
- `wx_openid`
- `wx_unionid`
- `wx_session_key_enc`：保留的可空字段，当前不持久化微信 session key。
- `nickname`
- `avatar_url`
- `phone`
- `role`
- `status`
- `last_login_at`
- `created_at`
- `updated_at`

当前登录流程完全不会写入 `wx_session_key_enc`，该字段通常保持 NULL，也不会保存微信 `session_key` 明文。openid 和 session_key 均不会暴露给小程序。

`User.role` 是小程序用户记录字段，不等同于后台管理员登录；后台权限以独立 `Admin` 表和 Flask session 为准。头像仅保存 `avatar_url` 字符串，当前后端没有文件或头像上传接口。

#### Device

数据库保存原始 `fault_message` 供内部诊断，但小程序摘要、详情和实时故障事件使用统一的公开文案：`UART_TIMEOUT` 映射为“CSI采集中断，请检查链路”，`NO_CSI_FRAME`/`NO_CSI_FRAME_TIMEOUT` 映射为“未收到CSI数据，请检查供电”，其他故障映射为“设备采集异常，请检查设备”。

- `id`

微信通知发送审计日志：

- `id`

模型关系与约束：

- `admin.username`、`users.wx_openid`、`devices.device_name` 均唯一。

### 3.4 数据库初始化与迁移边界

默认数据库为 `ESP32-MQTT-Backend-main/instance/app.db`，可用 `DATABASE_URL` 覆盖。SQLite 连接会启用外键约束。现有增量补列逻辑是面向旧 SQLite 的手写 `ALTER TABLE`；切换其他数据库前需要单独验证初始化与迁移行为。

## 4. WebSocket 实时事件设计

WebSocket 路由位于 `services/websocket_service.py`：

```text
/ws/v1/events?token=<access_token>
```

后端建立连接时：

1. 解析 token。
2. 校验用户存在且状态为 active。
3. 将 websocket 加入 `WebSocketHub` 中按用户分组的连接池。
4. 发送 `connection.ready`。
5. 循环等待客户端消息，连接关闭后移除。

推送消息结构：

```json
{
  "event": "device.runtime.changed",
  "device_name": "device001",
  "data": {}
}
```

事件按用户推送，只有设备所属用户会收到相关设备事件。

常见事件：

- `device.state.changed`：设备 online/offline/error 主状态变化。
- `device.runtime.changed`：硬件运行状态或检测状态变化。
- `device.fault`：设备进入故障。
- `detection.network-quality`：CSI 网络质量变化。
- `detection.fall-result`：算法识别到跌倒。

前端实际依赖的主要 `data` 字段：

| 事件 | `data` 关键字段 |
| --- | --- |
| `connection.ready` | `user_id` |
| `device.state.changed` | `state`、`last_seen_at` |
| `device.runtime.changed` | `runtime_state`、`detection_state`、`session`；控制回执还可能含 `control_ok`、`action`、`message`、`fault_cleared` |
| `device.fault` | `code`、公开 `message`、`template_data` |
| `detection.network-quality` | `session`、`network_quality`、`last_csi_at` |
| `detection.fall-result` | `fall_detected`、`fall_event_id`、`session`、`network_quality`、`occurred_at` |

小程序不会直接用 WebSocket payload 覆盖完整本地设备对象；多数事件只是触发重新拉取 REST 快照。只有控制失败提示、故障弹窗和跌倒跳转会直接读取上述部分字段。

## 5. MQTT 交互逻辑

### 5.1 MQTT 连接设计

MQTT 配置位于 `app/mqtt/config.py`。

默认配置：

```text
host     = 192.168.101.48
port     = 1883
username = csi_user
password = ""
keepalive = 60
```

可通过环境变量覆盖：

- `MQTT_BROKER_HOST` 或 `MQTT_HOST`
- `MQTT_BROKER_PORT` 或 `MQTT_PORT`
- `MQTT_BROKER_USERNAME` 或 `MQTT_USERNAME`
- `MQTT_BROKER_PASSWORD` 或 `MQTT_PASSWORD`
- `MQTT_KEEPALIVE`

后端使用 Paho MQTT。每个“已启用且已经由 `ensure_*` 激活”的设备拥有一个独立 `DeviceMqttClient`，client id 形如：

```text
python_<device_name>_001
```

`MqttManager` 管理所有设备客户端：

- `ensure_device(device_name)`：确保某设备 MQTT 客户端已创建并异步连接。
- `remove_device(device_name)`：移除并断开设备客户端。
- `publish_control(...)`：向设备下行控制主题发布控制命令。

`MQTT_AUTOSTART_DEVICES=true` 时会在服务启动阶段激活全部启用设备；默认值为 false 时，通常在小程序读取用户设备列表或详情后按需激活。禁用设备不会建立客户端。

如果 `MQTT_ENABLED` 为 false，后端不会真实连接 MQTT，控制设备时会返回 `MQTT_UNAVAILABLE`。

### 5.2 MQTT Topic 规范

topic 生成逻辑位于 `app/mqtt/topics.py`。

设备名规则：

```text
^[A-Za-z0-9_-]{1,32}$
```

统一前缀：

```text
csi/v1/devices
```

上行 topic：

```text
csi/v1/devices/{device_name}/up/online
csi/v1/devices/{device_name}/up/wifi
csi/v1/devices/{device_name}/up/status
csi/v1/devices/{device_name}/up/csi
csi/v1/devices/{device_name}/up/ack
csi/v1/devices/{device_name}/up/fault
```

下行控制 topic：

```text
csi/v1/devices/{device_name}/down/control
```

后端订阅采用精确 topic，不使用通配符。

QoS 设计：

| Topic 名称 | 方向 | QoS | 说明 |
| --- | --- | --- | --- |
| `online` | 上行 | 1 | 设备在线/离线 |
| `wifi` | 上行 | 1 | Wi-Fi 心跳或连通性 |
| `status` | 上行 | 1 | 硬件运行状态 |
| `csi` | 上行 | 0 | 高频 CSI 数据 |
| `ack` | 上行 | 1 | 控制命令回执 |
| `fault` | 上行 | 1 | 设备故障 |
| `control` | 下行 | 1 | start/stop/reset/clear_fault 控制 |

### 5.3 下行控制 payload

控制命令由 `DeviceMqttClient.publish_control()` 发布到：

```text
csi/v1/devices/{device_name}/down/control
```

通用 JSON：

```json
{
  "cmd": "control",
  "action": "start",
  "session": "sess-20260709-120000-abcdef"
}
```

支持 action：

- `start`
- `stop`
- `reset`
- `clear_fault`

其中：

- `start` 如果没有 session，后端会自动生成 session。
- `stop` 必须带当前 session。
- `reset` 和 `clear_fault` 可额外带 `command_id`、`reason`、`source`。

协调器会为控制流程生成内部 `command_id`，但当前真实 `DeviceMqttClient` 只在 `reset`/`clear_fault` JSON 中序列化 `command_id`、`reason`、`source`；`start`/`stop` 下行 JSON 只有 `cmd`、`action`、`session`。

复位命令示例：

```json
{
  "cmd": "control",
  "action": "reset",
  "session": "sess-xxx",
  "command_id": "cmd-xxx",
  "reason": "user_fault_confirm",
  "source": "user"
}
```

### 5.4 上行 online 处理

处理函数：`DeviceCoordinator._handle_online()`。

设备可上报：

```json
{
  "status": "online"
}
```

也兼容：

```json
{
  "online": true
}
```

或：

```json
{
  "state": "offline"
}
```

后端会将字符串、数字和布尔值统一解析为布尔在线状态。

在线时：

- 更新 `last_seen_at`。
- 更新 `last_online_at`。
- 如果设备不在故障中，将 `state` 置为 `online`。
- 推送 `device.state.changed`。

离线时：

- 如果设备正在检测，并且距离最后一次有效 CSI 或状态信号未超过硬超时，后端会忽略瞬时离线，避免硬件短暂抖动导致误判。
- 否则调用 `_mark_device_offline()`，清空检测 session、网络质量和故障状态。
- 推送 `device.state.changed`。

### 5.5 上行 wifi 处理

处理函数：`DeviceCoordinator._handle_wifi()`。

有效 payload：

```json
{
  "ok": true
}
```

如果 `ok` 为 true：

- 更新 `last_seen_at`。
- 如果设备不在故障中，置为 `online`。
- 更新 `last_online_at`。

### 5.6 上行 status 处理

处理函数：`DeviceCoordinator._handle_status()`。

硬件状态只接受：

- `booting`
- `idle`
- `uploading`
- `fault`

示例：

```json
{
  "state": "uploading",
  "session": "sess-xxx"
}
```

状态映射逻辑：

- `uploading`：表示硬件正在上传 CSI。已有当前 session 或上报了有效 session，且设备不处于 `stopping` 时，后端才会把检测状态推进到 `running`；双方都没有 session 时只更新硬件运行状态。
- `booting`：表示硬件重启中。后端会将检测状态重置为 `idle`，清空 session 和网络质量。
- `idle`：如果当前正在 starting/running/stopping，说明硬件已停止检测，后端会重置为 `idle` 并清空 session。
- `fault`：进入故障流程，记录故障码和故障消息。

如果设备已经处于故障状态，普通的非故障 status 不会覆盖故障，只更新时间戳。这可以避免硬件后续状态包把故障误清掉。

复位特殊逻辑：

- 用户发出 reset 后，后端记录 `_fault_reset_pending`。
- 如果之后收到 `state=idle` 且匹配 pending reset，后端会清除故障。
- 清除后推送 `device.runtime.changed`，并带上 `fault_cleared = true`。

### 5.7 上行 ack 处理

处理函数：`DeviceCoordinator._handle_ack()`。

控制回执示例：

```json
{
  "cmd": "control",
  "action": "start",
  "ok": true,
  "session": "sess-xxx",
  "state": "uploading",
  "err": 0,
  "msg": ""
}
```

后端会校验：

- `cmd` 省略时按 `control` 处理；显式提供时必须为 `control`。
- `action` 必须是 `start`、`stop`、`reset` 或 `clear_fault`。
- `ok` 必须能解析成布尔。
- 如果 ACK 和后端当前状态都带 session，二者必须一致，否则视为过期 ACK 并忽略；后端当前没有 session 时，不会仅因 ACK 自带 session 而拒绝它。

`start` ACK：

- 成功：`runtime_state` 设为 `uploading` 或 ACK 中合法状态，`detection_state` 设为 `running`。
- 失败：检测状态回到 `idle`，清空 session 和网络质量。

`stop` ACK：

- 成功：检测状态回到 `idle`，清空 session 和网络质量。
- 失败：检测状态回到 `running`。

`reset` 或 `clear_fault` ACK：

- 成功：调用 `_clear_fault()`，清除故障，设备回到 online + idle。
- 失败：保持 error/fault 状态。

ACK 处理完成后会推送 `device.runtime.changed`。

### 5.8 上行 fault 处理

处理函数：`DeviceCoordinator._handle_fault()` 和 `_record_fault()`。

故障上报示例：

```json
{
  "session": "sess-xxx",
  "code": "NO_CSI_FRAME_TIMEOUT",
  "msg": "启动检测后未收到 CSI 数据"
}
```

后端处理：

1. 校验 session，过期 session 的故障会被忽略。
2. 设置设备：
   - `state = error`
   - `runtime_state = fault`
   - `fault_code = code`
   - `fault_message = message`
3. 如果设备正在检测，将 `detection_state` 置为 `stopping`。
4. 清理当前 session 的 CSI 质量窗口和算法窗口。
5. 限流后推送 `device.fault`。
6. 对特定故障码尝试发送微信设备故障通知。
7. 如果当前存在检测 session、检测状态属于 `starting/running/stopping`，且该 session 尚未请求过自动停止，后端会发布一次 `stop` 控制命令，尝试让硬件停止上传。

会触发微信设备故障通知的故障码：

- `NO_CSI_FRAME`
- `NO_CSI_FRAME_TIMEOUT`
- `UART_TIMEOUT`

前端也会对这些故障码弹出“设备采集异常”提示。

### 5.9 上行 csi 处理

处理函数：`DeviceCoordinator._handle_csi()`。

CSI payload 解码位于 `services/csi_payload_service.py`。

当前支持格式：

```text
csib64-v2
```

JSON 外层字段包括：

```json
{
  "session": "sess-xxx",
  "fmt": "csib64-v2",
  "batch": 1,
  "frames": 45,
  "seq0": 1,
  "seq1": 45,
  "ts0": 1000000,
  "ts1": 2500000,
  "bytes": 12345,
  "data": "base64..."
}
```

`data` 是 Base64 编码后的二进制 CSIB 批次。二进制格式中包含：

- 批次头：magic、版本、帧数量、采样率、batch_no、起止时间戳。
- 每帧头：sequence、timestamp_us、rssi、first_word_invalid、CSI 长度。
- 每帧原始 CSI bytes。

后端会严格校验：

- magic 必须是 `CSIB`。
- 二进制版本必须匹配。
- JSON 中的帧数、batch、seq、ts、bytes 必须与二进制内容一致。
- 帧序号必须递增。
- 时间戳不能倒退。
- 单批最大帧数 80。
- 二进制最大 512 KB。
- 单帧 CSI 最大 384 bytes。

成功解码后：

1. 将每帧交错的 int8 I/Q 转换为子载波幅度，写入管理员热力图内存缓冲区。
2. 更新 `last_seen_at` 和 `last_csi_at`。
3. 设置 `runtime_state = uploading`。
4. 设置 `detection_state = running`。
5. 设置 `state = online`。
6. 交给 `CsiQualityTracker` 评估网络质量。
7. 将批次转为算法输入，放入内存滑动窗口。
8. 窗口帧数达到 `CSI_WINDOW_SIZE`、网络质量不为 `unknown`，且当前 `(device, session)` 尚未产生过跌倒事件时调用跌倒算法。
9. 如果算法返回 1，创建 `FallEvent`；同一 session 最多生成一次自动识别事件。
10. 推送 `detection.fall-result`。

当前 `services/fall_detect_service.py` 中的默认算法是安全占位实现：

```text
predict_fall(...) -> 0
```

也就是说，默认不会产生跌倒事件。项目通过 `app.config["FALL_PREDICTOR"]` 预留了算法替换入口，测试中会注入返回 1 的 predictor 来验证事件生成流程。

### 5.10 管理端实时 CSI 热力图

热力图链路由 `csi_feature_service.py`、`csi_live_buffer_service.py`、管理 API 和 `device_dashboard.js` 共同完成：

```text
已校验的 CsiFrame.raw_csi
  -> 按 [I0, Q0, I1, Q1, ...] 解析 int8
  -> amplitude = sqrt(I^2 + Q^2)
  -> 每设备/每 session 内存环形缓冲区
  -> GET /api/devices/{device_name}/csi-heatmap?frames=180
  -> 管理页面 Canvas 绘制
```

内存缓冲规则：

- 每台设备最多保存 300 帧，默认快照 180 帧；API 的 `frames` 会被限制在 30 到 300 之间。
- session 改变或子载波数量改变时重置该设备缓冲区。
- 设备进入故障，或 `DeviceCoordinator._clear_session()` 显式清理检测 session 时移除对应实时数据。
- 记录接收帧数、估算丢帧数、丢帧率、序号间隙、序号重置、FPS、最后序号、时间戳和 RSSI。

快照把所有幅度值的第 5/95 百分位作为显示上下界，归一化为 `0..255`，按“子载波行 × 时间列”的 row-major 布局编码为 `uint8`，再通过 Base64 返回。帧数不足时在矩阵左侧补零，最新帧位于最右列。

这套数据只存在于 Flask 进程内存：不会写入 SQLite，服务重启后消失，也不是历史 CSI 回放接口。没有实时帧时接口仍返回设备状态，但 `heatmap.ok=false`、`reason=no_live_csi`。

### 5.11 CSI 网络质量评估

质量评估位于 `services/csi_quality_service.py`。

后端按设备和 session 维护滚动窗口，使用这些指标判断质量：

- 批次间隔。
- 帧序号连续性。
- 丢帧率。
- invalid 帧比例。
- 是否出现非单调 batch 或 seq。
- 最大间隔是否超过软超时。

质量等级：

- `good`：丢帧和 invalid 很低，间隔稳定。
- `fair`：有轻微丢帧或间隔拉长，但仍可接受。
- `poor`：质量较差、间隔异常或数据问题明显。
- `unknown`：数据不足。

如果硬件重启导致 seq 回到很小值，后端会识别为 `seq_reset`，清空算法窗口，并将质量临时标为 `fair`，避免新旧数据混在一起。

### 5.12 离线与 CSI 超时监控

`DeviceCoordinator.start_monitor()` 会启动后台线程，每 2 秒扫描一次设备。

普通离线：

- 如果设备不在检测中，且 `last_seen_at` 超过 `OFFLINE_TIMEOUT_SECONDS`，标记为离线。

检测中 CSI 超时：

- 如果处于 starting/running/uploading，优先看 `last_csi_at`。
- 在启动宽限期内，不立刻报错。
- 超过软超时，降低网络质量。
- 超过恢复宽限期，继续保持 running，但质量为 poor。
- 超过硬超时，如果设备仍应处于检测状态，记录 `NO_CSI_FRAME_TIMEOUT` 故障。

这套设计避免因为短暂网络抖动把正在运行的设备直接踢成离线，同时又能在长时间无 CSI 时进入明确故障状态。

## 6. 核心业务闭环

### 6.1 用户登录闭环

```text
小程序 wx.login
  -> /api/v1/auth/wechat-login
  -> 后端 code2Session 换 openid
  -> 创建或更新 User
  -> 签发 access_token
  -> 小程序保存 token
  -> 拉取用户、设备、跌倒事件
  -> 建立 WebSocket
```

### 6.2 启动检测闭环

```text
用户点击开始检测
  -> 小程序请求订阅消息授权
  -> 小程序先显示正在启动并设置 pending 截止时间
  -> POST /api/v1/devices/{device}/control action=start
  -> DeviceCoordinator 校验设备在线、启用、无故障、空闲
  -> 生成 session 和后端内部 command_id
  -> MQTT 发布 down/control start
  -> 数据库标记 detection_state=starting
  -> 设备 ACK start ok 或 status uploading 或 CSI 到达
  -> 后端标记 running
  -> WebSocket 推送 device.runtime.changed
  -> 小程序刷新为检测运行中
```

### 6.3 停止检测闭环

```text
用户点击停止检测
  -> 小程序先显示正在停止并设置 pending 截止时间
  -> POST /api/v1/devices/{device}/control action=stop
  -> 后端校验当前 starting/running 且有 session
  -> MQTT 发布 down/control stop
  -> 数据库标记 detection_state=stopping
  -> 设备 ACK stop ok 或 status idle
  -> 后端清空 session，状态回到 idle
  -> WebSocket 推送
  -> 小程序刷新
```

### 6.4 跌倒事件闭环

```text
设备上传 CSI
  -> 后端解码 csib64-v2
  -> 更新设备 last_csi_at 和网络质量
  -> CSI 批次进入内存算法窗口
  -> predictor 返回 1
  -> 创建 FallEvent(status=pending)
  -> WebSocket 推送 detection.fall-result
  -> 小程序跳转跌倒告警页
  -> 用户确认安全
  -> PATCH /api/v1/fall-events/{id}
  -> 后端标记 confirmed，写 handled_at
```

注意：代码中微信跌倒通知服务 `send_fall_alert()` 已实现，并且管理端模拟跌倒会调用它；CSI 自动识别跌倒路径当前会落库并推 WebSocket，但在 `device_state_service.py` 中没有直接调用 `send_fall_alert()`。

### 6.5 故障处理与保留的复位链路

```text
设备上报 fault 或后端检测 CSI 硬超时
  -> 后端记录 fault_code/fault_message
  -> 设备 state=error runtime_state=fault
  -> 如果有运行 session，自动 MQTT 发布 stop
  -> WebSocket 推送 device.fault
  -> 对指定 CSI 故障码尝试微信设备故障通知
  -> 小程序弹出采集异常
  -> 设备详情显示异常 banner 并禁用检测控制
```

后端仍支持以下恢复链路，但当前小程序没有按钮触发：

```text
POST /api/v1/devices/{device}/reset-fault
  -> MQTT 发布 reset
  -> 设备 ACK reset ok 或 status idle
  -> 后端清除故障并恢复 online/idle
  -> WebSocket 推送 device.runtime.changed
```

## 8. 数据一致性与容错设计

### 8.1 设备状态以数据库为准

前端展示的设备状态来自后端数据库快照，而不是直接依赖 MQTT 消息。MQTT 只负责驱动后端更新快照。

这样做的好处：

- 小程序刷新后仍能看到最近状态。
- WebSocket 丢消息后可以重新拉取 REST 接口恢复状态。
- 后台管理端和小程序看到的是同一套数据。

### 8.2 控制请求不是最终状态

控制接口返回 202 只表示后端已接收并尝试发布 MQTT 命令。硬件最终是否执行成功，取决于：

- `ack` 回执。
- `status` 上报。
- `csi` 是否开始到达。
- 后端超时扫描结果。

所以前端采用“乐观 pending 状态 + WebSocket + 短轮询确认”的组合。

### 8.3 Session 防串扰

检测任务通过 `current_session` 绑定。

后端会忽略：

- 过期 session 的 CSI。
- 过期 session 的 ACK。
- 过期 session 的 fault。
- 上传中报告了不一致 session 的 status。

这可以避免硬件重启、旧包延迟或重复发布影响当前任务。

### 8.4 原始 CSI 不落库

CSI 原始数据只在内存中短暂存在，用于质量评估、跌倒算法窗口和管理员实时热力图。热力图缓冲区每设备最多 300 帧，并随 session 切换、故障清理或进程重启而丢失。数据库只保存：

- 设备最新状态。
- 网络质量等级。
- 跌倒事件结果。
- 通知日志。

这减少了数据库压力，也避免 SQLite 承担高频大体量 CSI 存储。

- `fall_detect_service.predict_fall()` 默认始终返回 0，正式跌倒识别需要注入真实模型或算法。
- CSI 自动识别跌倒路径会落库并推送 WebSocket，但没有调用已经实现的 `send_fall_alert()`；目前只有管理端模拟跌倒会走微信跌倒通知。
- 小程序没有单条跌倒事件详情接口，告警页会拉取最多 100 条后按 id 查找；首页也只检查最新 1 条事件是否待处理。
- 设备详情页只在最近 20 条事件中查找本设备待处理告警，也可能漏掉更早记录。
- 后端故障复位接口和 MQTT 链路仍在，但设备详情页没有复位入口；需要明确恢复应由管理员、自动流程还是用户触发，再补 UI。
- “我的”页手机号和独立订阅入口当前隐藏；设备详情的两个运行字段也都显示检测状态，需按产品要求决定是否恢复和修正。
- access token 没有 refresh token 流程，`expires_in` 目前也未用于提前续期。正常使用中遇到 401 时只会清除 token/认证态，不会由请求封装自动重新登录，也不会显式关闭已有 WebSocket；当前界面还没有退出登录入口。
- 管理员 CSI 热力图只保存在单个 Flask 进程内存中，没有历史数据；如果未来采用多 worker 部署，需要引入共享实时缓冲或保证 MQTT 摄入与热力图请求落在同一进程。
- 管理员删除设备时当前只删除数据库记录并移除 MQTT 客户端，没有同步清除该设备名的热力图缓冲；若很快用同名设备重建，可能短暂读到旧内存帧，应在删除路径补 `clear_device()`。
- production 的 `apiBaseUrl` 和 `wsBaseUrl` 为空，`env.ts` 又没有受版本管理的样板，应补一份不含密钥的配置模板。
- 当前数据库升级只覆盖四个通知列，不是完整迁移系统；模型结构继续演进前应接入 Alembic 或等效迁移方案。
- 生产安全需要处理强制 debug、开发默认密钥、未启用 `Secure` 的 session cookie、WebSocket query token、MQTT 明文/无 TLS 等问题。
- `instance/` 不进入版本库，但 SQLite 中包含 openid、手机号和通知记录；部署与迁移时需要单独设置文件权限、备份和数据保留策略。
- 当前没有 CI 或覆盖率配置，且默认 pytest 不收集 MQTT 测试目录，应补统一的自动化验证入口。
- `csi_miniprogram--/fall_detector` 使用已删除依赖和旧协议，当前不可作为运行模块；应迁移到正式后端协议或移除，避免与 `services/fall_detect_service.py` 混淆。
- 如果以后需要长期保存原始 CSI，建议使用独立的时序存储或文件/对象存储，不要直接写入当前 SQLite 主库。
