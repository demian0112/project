# 项目交接文档 — CSI 跌倒检测系统全栈开发指南

## 一句话概括

这是一个 **WiFi CSI 跌倒检测智能守护系统**。用户在家放置 ESP32-S3 设备组（A/B/C 三块板），通过微信小程序远程开关 CSI 监测，后端通过 MQTT 传输控制信号，跌倒检测模块分析 WiFi 信号变化判断是否有人跌倒，检测到异常后小程序弹出告警。

**当前进度**：小程序→后端→Mosquitto 的控制链路已打通（第二段验收过），跌倒告警 MQTT 接收链路已实现，跌倒检测模块（fall_detector）和 C 板控制下行尚未实现。

## 3. 后端详解

### 3.1 依赖关系图

```
config.py       ← 零依赖，纯常量。改了全项目生效
    │
    ▼
models.py       ← 仅依赖 pydantic。定义所有数据结构
    │
    ├──────────┬──────────────┐
    ▼          ▼              ▼
device_state  mqtt_client   (独立模块)
    │          │
    │          ▼
    │        control.py      ← 依赖 models + mqtt_client
    │          │
    ▼          ▼
main.py         ← 依赖以上全部。只做路由注册 + 启动逻辑
```

**设计原则**：每层只依赖下一层，不跨层调用。任何一个模块都可以单独替换而不影响其他模块。

### 3.2 config.py — 所有可变配置的唯一入口

```python
DEVICE_ID = "esp32s3_c_csi_2s_001"        # 与 C 板 device_id 严格一致

MQTT_HOST = "127.0.0.1"                   # 同机→127.0.0.1；跨机→局域网IP；上云→公网IP
MQTT_PORT = 1883
MQTT_USERNAME = "esp32"
MQTT_PASSWORD = "esp32pass"
MQTT_CLIENT_ID = "backend_control_001"

MQTT_TOPIC_CONTROL = "esp32s3/control"     # 后端→C板 控制命令
MQTT_TOPIC_STATUS = "esp32s3/status"       # C板→后端 状态回执
MQTT_TOPIC_FALL_ALERT = "esp32s3/fall_alert"  # fall_detector→后端 跌倒告警

MQTT_ENABLED = True   # False 时不连 MQTT，HTTP API 继续正常服务
```

**换环境时只改这个文件**。

### 3.3 models.py — 前后端数据契约

所有 API 请求/响应格式在此定义，前后端必须一一对应。

| 模型 | 方向 | 用途 | 关键字段 |
|---|---|---|---|
| `SwitchRequest` | 小程序→后端 | 开关请求 | `device_id`, `enable`, `source` |
| `DeviceState` | 后端内部 | 设备状态 | `monitor_enabled`, `state_text` |
| `DeviceCommand` | 后端→MQTT | 控制命令 | `cmd`, `enable`, `target`, `request_id` |
| `FallAlert` | MQTT→后端 | 跌倒告警 | `detected`, `confidence`, `duration_seconds` |
| `HealthResponse` | 后端→小程序 | 健康检查 | `ok`, `message`, `time` |
| `DeviceStateResponse` | 后端→小程序 | 状态查询 | `ok`, `device_state` |
| `SwitchResponse` | 后端→小程序 | 开关结果 | `ok`, `request_id`, `reserved_device_command` |
| `FallAlertResponse` | 后端→小程序 | 告警查询 | `ok`, `alert` (可为 null) |

**新增接口时**：先在 models.py 加模型 → 再在 main.py 加路由 → 最后在 api.ts 加类型和请求方法。

### 3.4 device_state.py — 设备状态存储

```python
get_state() → DeviceState       # 读取当前状态
update_state(device_id, enable, request_id, source) → DeviceState  # 更新并返回
```

当前用**内存字典**存储单设备状态。单进程重启后状态丢失。

**替换方案**：若需持久化或多设备，改此文件内部实现（换成 Redis / SQLite / 文件），`main.py` 调用方零改动。

**已知问题**：`_store` 初始化 device_id 为 `"esp32s3_csi_001"`（旧值），与 config.py 中 `"esp32s3_c_csi_2s_001"` 不一致。首次 `update_state()` 调用后会被覆盖，不影响功能，但建议改为从 config 读取（低优先级）。

### 3.5 mqtt_client.py — MQTT 客户端封装

```
start()                     # 连接 Broker + 订阅 status 和 fall_alert
stop()                      # 断开连接
publish_command(cmd) → dict # 发布控制命令到 esp32s3/control
get_latest_fall_alert() → dict | None  # 获取最新缓存的跌倒告警
is_connected() → bool
```

**降级设计**：

| 条件 | 行为 |
|---|---|
| `config.MQTT_ENABLED = False` | 不连接 MQTT，HTTP API 正常服务 |
| `paho-mqtt` 未安装 | 打印 warning，跳过 |
| Broker 不可达 | 记录 error，HTTP API 不受影响 |
| `publish_command()` 在 MQTT 未启用时 | 返回 `{mqtt_publish_rc: None}`，不抛异常 |

**消息处理逻辑**：`_on_message` 回调中，如果收到 `esp32s3/fall_alert` 的消息，解析 JSON 后缓存到 `_latest_fall_alert`，供 `GET /api/device/fall` 查询。收到 `esp32s3/status` 的消息仅打印日志（状态回执逻辑待实现）。

### 3.6 control.py — 命令生成

```python
generate_and_publish(device_id, enable, request_id?, source?) → dict
```

生成的命令格式：
```json
{
  "cmd": "set_monitor",
  "enable": true,
  "target": "esp32s3_c_csi_2s_001",
  "request_id": "d68b3777-...",
  "source": "backend",
  "time": "2026-06-29T18:28:45"
}
```

独立于 HTTP 层，可被任何调用方使用（API、脚本、CLI 测试）。

### 3.7 main.py — FastAPI 入口

| 方法 | 路径 | 说明 | 响应模型 |
|---|---|---|---|
| GET | `/api/health` | 健康检查 | `HealthResponse` |
| GET | `/api/device/state` | 获取设备状态 | `DeviceStateResponse` |
| POST | `/api/device/switch` | 发送开关命令 | `SwitchResponse` |
| GET | `/api/device/fall` | 查询最新跌倒告警 | `FallAlertResponse` |

**启动**：
```bat
cd backend
pip install fastapi uvicorn paho-mqtt
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

**启动流程**：
1. `on_startup` → `mqtt_client.start()` → 连接 MQTT + 订阅 `esp32s3/status` 和 `esp32s3/fall_alert`
2. `on_shutdown` → `mqtt_client.stop()`

**CORS**：开发阶段允许所有来源（`allow_origins=["*"]`），上线后改为具体域名。

### 4.3 api.ts — 请求封装层

**这是小程序与后端之间的唯一通信层，所有 `wx.request` 集中在此**。

```typescript
const BASE_URL = 'http://127.0.0.1:8000'   // ← 唯一需要改的地址

// 暴露 4 个方法，一一对应后端 4 个 API：
checkHealth()       → Promise<HealthResponse>

```

**类型定义**与 `backend/models.py` 一一对应，改任何接口都要两边同步。

### 4.4 index 页面 — 核心监测页

**UI 设计**：白色医疗健康风

- **Hero Card**：顶部大卡片，包含实时状态标签（绿色 pill）、Canvas 环形仪表盘（`gauge-ring` 组件）、模拟信号波动画
- **告警态**：检测到跌倒时卡片切换红色主题、环形仪表盘变红、雷达图标切换为跌倒小人图标
- **告警横幅**：红色 banner，显示置信度 + 持续时间 + 时间戳
- **操作按钮**："确认安全"清除告警，"联系家人"（功能占位）
- **微信通知提示**：绿色微信图标 + 已通知提示
- **摘要卡片**：CSI 信号质量 + 今日守护时长（当前为占位静态数据）
- **设备卡片**：设备名称、型号、在线状态
- **电源按钮**：底部圆形开关按钮

**轮询机制**：`onShow()` 时启动 2 秒间隔轮询 `getFallAlert()`，`onHide()` 时停止。检测到告警后更新页面所有相关状态，告警清除后自动恢复。

### 4.5 gauge-ring 组件

用 **Canvas 2D** 绘制弧形进度环（因为 WXSS 不支持 `conic-gradient`）。属性：`progress`(0-1)、`color`、`bgColor`、`lineWidth`。中心通过 `<slot>` 插入自定义内容。

### 4.6 custom-tab-bar

自定义底部导航栏，包含"监测"和"我的"两个 tab。`switchTab` 方法通过 `wx.switchTab()` 切换。

## 5. 数据流全链路

### 5.1 开关控制流（已打通验收）

```
小程序点击开关
    │
    ▼
api.ts → POST /api/device/switch  {device_id, enable: true, source: "wechat_miniprogram"}
    │
    ▼
main.py → set_device_switch()
    ├── 生成 request_id (UUID)
    ├── device_state.update_state()      ← 写入内存: monitor_enabled=True, state_text="monitoring"
    ├── control.generate_and_publish()   ← 生成 {"cmd":"set_monitor","enable":true,...}
    └── mqtt_client.publish_command()    ← publish 到 esp32s3/control
            │
            ▼
        Mosquitto Broker → (预留)C板订阅
```

验收通过的标准：`mosquitto_sub -t esp32s3/#` 能看到 `esp32s3/control {"cmd":"set_monitor","enable":true,...}`。

### 5.2 跌倒告警流（接收链路已实现，检测模块待开发）

```
CSI数据 → C板 → MQTT esp32s3/test → fall_detector(待开发)
                                          │
                                    模型推理判断跌倒
                                          │
                                    publish 到 esp32s3/fall_alert
                                          │
                                          ▼
mqtt_client._on_message() 收到 → 缓存到 _latest_fall_alert
                                          │
                                          ▼
小程序轮询 GET /api/device/fall → 返回 FallAlert
                                          │
                                          ▼
                               index.ts 更新 isAlert=true
                               UI 切换红色告警态
```

### 5.3 状态回执流（预留，C 板待实现）

```
C 板收到控制命令并执行
    │
    ▼
C 板 publish 到 esp32s3/status
    │  {"device_id":"...","request_id":"...","monitor_enabled":true,"state":"monitoring","ack":true}
    ▼
mqtt_client._on_message() 收到 → (当前仅打印日志，状态更新逻辑待实现)
```

### 7.2 ESP32-S3 设备组（外部硬件系统）

详细文档见：`readme/ESP32-S3 CSI 数据采集、Mosquitto 上传与 Python 可视化系统阶段性总结(1).txt`

**A 板（AP 发射端）**：创建 WiFi `CSI_AP`，密码 `12345678`，Channel 6，IP `192.168.4.1`。

**B 板（CSI 采集板）**：连接 `CSI_AP`，ping A 板获取 CSI（约 30 Hz），通过 UART1 (GPIO17 TX→GPIO18 RX, 921600 baud) 发送给 C 板。

**C 板（MQTT 网关板）**：连接路由器 2.4G WiFi，UART 接收 B 板 CSI 帧，2 秒打包为 CSIB 二进制→Base64→JSON，发布到 `esp32s3/test`。MQTT 参数：`mqtt://192.168.101.48:1883`，Client ID `esp32s3_c_csi_2s_001`。

**C 板待实现**：
1. 订阅 `esp32s3/control` 并解析 `set_monitor` 命令
2. 通过 UART 发送 `START_CSI` / `STOP_CSI` 到 B 板
3. 执行后回复 `esp32s3/status` 状态回执

### 7.3 Python CSI 可视化程序

`readme/csi_mqtt_viewer_sliding_33fps(1).py`：订阅 `esp32s3/test`，解析 CSI 数据，PySide6 + pyqtgraph 以 33 FPS 滑动热力图实时显示 CSI 幅度。

## 10. 常见问题排查

| 现象 | 可能原因 | 解决 |
|---|---|---|
| 后端 `MQTT connection failed: timed out` | `MQTT_HOST` IP 不对或 Mosquitto 未启动 | 确认窗口1 Mosquitto 在跑，检查 `config.py` 中 `MQTT_HOST` 是否为 `127.0.0.1` |
| 小程序 `request:fail url not in domain list` | 域名校验未关闭 | 微信开发者工具 → 详情 → 勾选"不校验合法域名" |
| `mosquitto_sub` 连不上 `Error: 连接尝试失败` | IP 错误 | 与后端同机用 `127.0.0.1`，跨机用局域网 IP |
| 点开关后 MQTTX 无消息 | 订阅 Topic 不对 | 确认订阅的是 `esp32s3/#`，不是 `home/garden/fountain` |
| 小程序 `Component does not have a method` | 使用了 `Component()` 写法 | 改用 `Page()` 写法，事件处理方法放在第一层 |
| `ModuleNotFoundError: No module named 'paho'` | 未安装 MQTT 库 | `pip install paho-mqtt` |
| Mosquitto `Error: Unable to open config file` | 配置文件路径错误 | 确认路径 `C:\Program Files\Mosquitto\mosquitto_esp32.conf` 存在 |
| C 板 `Header invalid, resync` | B 板与 C 板 UART 协议版本不匹配 | B 板重烧正确版本固件 |
| JSON buffer 申请失败 (C 板) | ESP32 内存不足 | 改为动态申请 |
