# 微信小程序开关控制 + Mosquitto 验收层阶段性成果

## 1. 项目定位

本项目在已有 ESP32-S3 CSI 采集系统（A 板 AP → B 板 CSI 采集 → C 板 MQTT 上传 → Python 可视化）的基础上，新增**微信小程序远程开关控制能力**，实现：

```
微信小程序开关 → FastAPI 后端 → Mosquitto → C 板 → 控制 B 板 CSI 采集启停
```

当前阶段已打通：**小程序 → 后端 → Mosquitto**（第二阶段验收通过）。

## 4. 后端模块详解

### 4.1 模块依赖关系

```
config.py (零依赖，纯常量)
    ↓
models.py (仅依赖 pydantic)
    ↓
device_state.py (依赖 models)    mqtt_client.py (依赖 config)
    ↓                              ↓
control.py (依赖 models + mqtt_client)
    ↓
main.py (依赖以上全部，只做路由 + 启动)
```

每个模块职责单一，可独立测试，可独立替换。

### 4.2 config.py — 全局配置

```python
# 设备信息
DEVICE_ID = "esp32s3_c_csi_2s_001"       # 与 C 板 device_id 一致

# MQTT Broker
MQTT_HOST = "127.0.0.1"                   # 同机部署用 127.0.0.1；跨机改为实际 IP
MQTT_PORT = 1883
MQTT_USERNAME = "esp32"
MQTT_PASSWORD = "esp32pass"
MQTT_CLIENT_ID = "backend_control_001"

# MQTT Topics（与 C 板约定）
MQTT_TOPIC_CONTROL = "esp32s3/control"    # 后端 → C 板：控制命令
MQTT_TOPIC_STATUS = "esp32s3/status"      # C 板 → 后端：状态回执

# 功能开关
MQTT_ENABLED = True                       # False 时后端不连 MQTT，仍可接收 HTTP 请求
```

**设计意图**：所有环境差异集中在此文件。上云时只改 `MQTT_HOST` 和 `MQTT_PORT`，其他文件不动。

### 4.3 models.py — 数据模型

定义了前端 ↔ 后端之间完整的数据契约：

| 模型 | 方向 | 用途 |
|---|---|---|
| `SwitchRequest` | 小程序 → 后端 | `{device_id, enable, source}` |
| `DeviceState` | 后端内存 | 设备状态快照 |
| `DeviceCommand` | 后端 → Mosquitto | 控制命令格式 |
| `SwitchResponse` | 后端 → 小程序 | 含完整状态 + 命令回显 |
| `HealthResponse` | 后端 → 小程序 | 健康检查 |
| `DeviceStateResponse` | 后端 → 小程序 | 状态查询 |

### 4.4 device_state.py — 状态存储

当前用内存字典存储单设备状态。对外暴露两个函数：

```python
get_state() → DeviceState       # 读取当前状态
update_state(...) → DeviceState # 更新状态并返回快照
```

**替换方案**：若需持久化或多设备，只需改此文件内部实现（如接入 Redis / SQLite），调用方 `main.py` 无需改动。

### 4.5 mqtt_client.py — MQTT 封装

封装了 MQTT 连接、发布、订阅的全部逻辑。

**降级策略**：
- `config.MQTT_ENABLED = False` → 所有操作安全降级为空操作，不抛异常
- `paho-mqtt` 未安装 → 打印 warning，跳过连接
- Broker 不可达 → 记录 error，不影响 HTTP 服务正常运行

对外暴露四个函数：

```python
start()                             # 启动连接（main.py startup 时调用）
stop()                              # 断开连接（main.py shutdown 时调用）
publish_command(cmd: DeviceCommand) # 发布控制命令到 esp32s3/control
is_connected() → bool               # 查询连接状态
```

### 4.6 control.py — 命令生成

独立于 HTTP 层的纯逻辑模块，可被任何调用方使用（API、脚本、测试）。

```python
generate_and_publish(
    device_id: str,
    enable: bool,
    request_id: str | None = None,
    source: str = "backend",
) → dict
```

生成的控制命令格式（与 C 板约定的接口）：

```json
{
  "cmd": "set_monitor",
  "enable": true,
  "target": "esp32s3_c_csi_2s_001",
  "request_id": "b3a8f...",
  "source": "backend",
  "time": "2026-06-29T18:28:45"
}
```

### 4.7 main.py — 路由入口

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/health` | 健康检查，返回 `{ok, message, time}` |
| GET | `/api/device/state` | 获取设备状态，返回 `{ok, device_state}` |
| POST | `/api/device/switch` | 接收开关信号，请求体 `{device_id, enable, source}`，返回完整状态 + 生成的命令 |

启动命令：

```bat
cd backend
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

## 7. 与 ESP32-S3 现有系统的对接

### 7.1 一致的部分

| 项目 | 后端/小程序 | ESP32-S3 C 板 | 状态 |
|---|---|---|---|
| Broker 地址 | `config.py` → `MQTT_HOST` | `192.168.101.48:1883` | ✅ 一致 |
| 用户名/密码 | `esp32` / `esp32pass` | `esp32` / `esp32pass` | ✅ 一致 |
| Topic control | `esp32s3/control` | `MQTT_TOPIC_CONTROL` | ✅ 一致 |
| Topic status | `esp32s3/status` | `MQTT_TOPIC_STATUS` | ✅ 一致 |
| device_id | `esp32s3_c_csi_2s_001` | `esp32s3_c_csi_2s_001` | ✅ 一致 |

### 7.2 C 板待实现的功能

当前 C 板只有**数据上行**（CSI 采集 → MQTT 上传），尚未实现**控制下行**。后续 C 板需增加：

1. **订阅 `esp32s3/control`**：接收后端发来的 `set_monitor` 命令
2. **解析命令**：比较 `target` 字段与自身 `device_id`，匹配后执行
3. **UART 控制 B 板**：通过 UART 发送 `START_CSI` / `STOP_CSI` 指令
4. **状态回执**：执行后 publish 到 `esp32s3/status`：
   ```json
   {
     "device_id": "esp32s3_c_csi_2s_001",
     "request_id": "xxx-xxx-xxx",
     "monitor_enabled": true,
     "state": "monitoring",
     "ack": true
   }
   ```

### 7.3 终端用户侧的改动

接入控制功能后，`csi_mqtt_viewer_sliding_33fps.py` 需要忽略 `esp32s3/status` 和 `esp32s3/control` 类型的消息（代码已有此逻辑：遇到 `type != "csi_batch"` 自动跳过）。

## 9. 配置开关说明

`backend/config.py` 中的 `MQTT_ENABLED`：

| 值 | 行为 | 适用场景 |
|---|---|---|
| `True` | 后端启动时连接 Mosquitto，发布控制命令 | Mosquitto 已就绪 |
| `False` | 后端不连 Mosquitto，仅接收 HTTP 请求并返回生成的命令 | Mosquitto 未安装或调试后端 API 时 |

无论 `MQTT_ENABLED` 设为何值，HTTP API 始终正常工作，不会因 MQTT 不可用而崩溃。

## 10. 关键设计决策

1. **配置与逻辑分离**：所有可变参数集中在 `config.py` 和 `api.ts`，业务模块零硬编码
2. **MQTT 可选降级**：MQTT 不可用时后端 HTTP 服务不受影响
3. **数据契约显式化**：Pydantic 模型和 TypeScript 接口一一对应，前后端类型同步
4. **单文件职责**：每个模块只做一件事，可独立测试和替换
5. **device_id 全链路统一**：小程序 → 后端 → MQTT → C 板，使用同一个 `esp32s3_c_csi_2s_001`
6. **request_id 全链路追踪**：每次开关操作生成 UUID，贯穿后端日志、MQTT 命令、设备回执
