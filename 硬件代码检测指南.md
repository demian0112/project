# 硬件代码检测指南

本文用于联调排查微信小程序、Python 后端和 ESP32 硬件之间的业务闭环。硬件协议以 `hardware代码交互逻辑说明.md` 为准，当前正式后端为 `ESP32-MQTT-Backend-main`，小程序为 `csi_miniprogram--/miniprogram`。

核心原则：

- Python 后端不直接操作 UART，只通过 MQTT 控制 C 板。
- C 板负责 MQTT、配网、控制 B 板、上传状态和 CSI。
- B 板只负责接收 A 板触发的 CSI，并通过 UART 二进制帧发给 C 板。
- 小程序只调用 Python HTTP/WebSocket，不直接接触 MQTT 或硬件。

## 0. 联调前基础检查

### 0.1 服务地址

小程序配置：

```text
csi_miniprogram--/miniprogram/config/env.ts
apiBaseUrl = http://后端IP:5000
wsBaseUrl  = ws://后端IP:5000/ws/v1/events
```

后端配置：

```text
ESP32-MQTT-Backend-main/.env
MQTT_ENABLED=1
MQTT_HOST=Broker IP
MQTT_PORT=1883
MQTT_USERNAME=和硬件一致
MQTT_PASSWORD=和硬件一致
```

硬件配置：

```text
C板 MQTT Broker、端口、用户名、密码必须和后端一致
C板 device_name 必须和后台 devices.device_name 一致
```

### 0.2 Topic 总表

硬件必须使用下面这套 v1 Topic：

```text
C板 -> MQTT: csi/v1/devices/{device_name}/up/online
C板 -> MQTT: csi/v1/devices/{device_name}/up/wifi
C板 -> MQTT: csi/v1/devices/{device_name}/up/status
C板 -> MQTT: csi/v1/devices/{device_name}/up/csi
C板 -> MQTT: csi/v1/devices/{device_name}/up/ack
C板 -> MQTT: csi/v1/devices/{device_name}/up/fault
MQTT -> C板: csi/v1/devices/{device_name}/down/control
```

如果硬件仍在使用旧 topic，例如 `esp32s3/control`、`esp32s3/status`、`esp32s3/test`，当前正式 Flask 后端不会处理。

### 0.3 快速监听

用 MQTT 客户端或 Mosquitto 工具监听某个设备：

```powershell
mosquitto_sub -h BrokerIP -p 1883 -u 用户名 -P 密码 -t "csi/v1/devices/esp01/#" -v
```

观察后端发控制命令：

```powershell
mosquitto_sub -h BrokerIP -p 1883 -u 用户名 -P 密码 -t "csi/v1/devices/esp01/down/control" -v
```

## 1. 业务一：微信用户首次登录与手机号

### 小程序动作

1. 用户首次打开小程序。
2. 小程序调用 `wx.login()` 获取一次性 code。
3. 小程序请求 Python 后端 `POST /api/v1/auth/wechat-login`。
4. 用户可在个人中心授权手机号，小程序请求 `POST /api/v1/me/phone`。

### Python 后端动作

1. 后端用 code 调微信接口换取 `openid`。
2. 后端按 `openid` 查询 `users`。
3. 不存在则创建用户，存在则更新 `last_login_at`。
4. 手机号授权成功后写入 `users.phone`。
5. 后端签发 access token，小程序后续请求携带 Bearer Token。

### 硬件应该做什么

本业务阶段硬件不需要动作。

硬件排查重点：

- 不要把微信登录失败误判为硬件问题。
- 如果小程序登录失败，优先检查后端 `WECHAT_APPID`、`WECHAT_SECRET`、小程序 AppID 是否匹配。
- 此阶段不会出现 MQTT topic，也不会改变 `devices` 状态。

## 2. 业务二：管理员绑定设备到用户

### 小程序动作

1. 用户完成登录后，小程序可访问 `/api/v1/devices`。
2. 如果管理员尚未绑定设备，小程序显示暂无设备。
3. 管理员绑定后，用户下次进入或刷新即可看到自己的设备。

### Python 后端动作

1. 管理员后台创建或编辑 `devices`。
2. `devices.owner_user_id` 指向具体微信用户。
3. `devices.device_name` 是后端与硬件通信的唯一设备名。
4. 小程序只返回当前登录用户名下的设备。

### 硬件应该做什么

硬件必须保证 C 板 `device_name` 与后台 `devices.device_name` 完全一致。

示例：

```text
后台 device_name: esp01
C板 device_name: esp01
Python 订阅: csi/v1/devices/esp01/up/...
Python 下发: csi/v1/devices/esp01/down/control
```

如果不一致：

- 小程序能看到设备，但状态可能一直离线。
- 后端可能给 `esp01` 下发 control，C 板实际订阅的是另一个设备名，硬件不会响应。
- Broker 上可能能看到硬件消息，但 Python 后端不会处理到对应数据库记录。

检查点：

```text
C板 MQTT Client ID 应为 esp_{device_name}_001
Python MQTT Client ID 应为 python_{device_name}_001
```

## 3. 业务三：设备在线、离线、错误状态显示

设备状态分为三类：

```text
online  在线
offline 离线
error   错误
```

检测运行状态另有一组字段：

```text
runtime_state: booting / idle / uploading / fault
detection_state: idle / starting / running / stopping
```

### 3.1 C板上线

#### 小程序动作

1. 用户进入首页或设备详情页。
2. 小程序请求 `/api/v1/devices` 或 `/api/v1/devices/{device_name}`。
3. 小程序通过 WebSocket 或轮询刷新设备状态。

#### Python 后端动作

1. 后端为数据库中启用的设备创建 MQTT 客户端。
2. 后端订阅该设备的六个上行 topic。
3. 收到 `up/online` 或 `up/status` 后更新 `devices` 表。

#### 硬件应该做什么

C 板连接 Wi-Fi 和 MQTT 成功后，必须发布：

```text
topic: csi/v1/devices/{device_name}/up/online
qos: 1
retain: true
payload:
{
  "status": "online",
  "ip": "192.168.1.122",
  "fw": "csi-softap-mqtt-1.0.0",
  "ts": 240
}
```

随后发布 Wi-Fi 信息：

```text
topic: csi/v1/devices/{device_name}/up/wifi
qos: 1
retain: true
payload:
{
  "ok": true,
  "ip": "192.168.1.122",
  "rssi": -45,
  "ssid": "your_wifi_ssid",
  "ts": 240
}
```

随后开始周期发布 status：

```text
topic: csi/v1/devices/{device_name}/up/status
qos: 1
retain: false
payload:
{
  "state": "idle",
  "session": "",
  "uart": true,
  "upload": false,
  "rssi": -45,
  "heap": 158720,
  "uptime": 240000,
  "ts": 240
}
```

排查判断：

- Broker 能看到 `up/online`，但小程序仍离线：检查 `device_name` 是否和数据库一致。
- 后端启动晚于硬件：`up/online` 必须 retain=true，否则后端启动后可能收不到上线状态。
- 小程序显示在线但无法启动：检查后端是否收到最近一次 `up/status`，因为启动前会校验 status 是否超时。

### 3.2 C板离线

#### Python 后端动作

1. 收到 `up/online {"status":"offline"}` 后，将设备状态改为 `offline`。
2. 清空当前 session。
3. 将检测状态收敛为 `idle`。
4. 通过 WebSocket 通知小程序刷新。

#### 硬件应该做什么

C 板 MQTT 连接时必须配置 Last Will：

```text
topic: csi/v1/devices/{device_name}/up/online
qos: 1
retain: true
payload:
{
  "status": "offline",
  "reason": "lwt"
}
```

正常主动断开前也建议发布同样的 offline payload。

排查判断：

- 断电后小程序迟迟不离线：检查 C 板是否配置 LWT，Broker 是否代发 offline。
- offline payload 没有 retain=true：后端重启后可能拿不到最后离线状态。

### 3.3 C板错误状态

#### Python 后端动作

1. 收到 `up/fault` 或 `up/status state=fault`。
2. 将 `devices.state` 改为 `error`。
3. 将 `runtime_state` 改为 `fault`。
4. 保存 `fault_code` 和 `fault_message`。
5. 如果当前有 session，后端自动下发 stop。
6. WebSocket 通知小程序显示异常。

#### 硬件应该做什么

如果 C 板处于 `uploading`，但超过 `UART_TIMEOUT_MS` 没有收到 B 板 CSI，必须发布：

```text
topic: csi/v1/devices/{device_name}/up/fault
qos: 1
retain: false
payload:
{
  "code": "UART_TIMEOUT",
  "msg": "no csi frame from b board",
  "state": "fault",
  "ts": 300
}
```

如果打包 CSI 或构造 JSON 内存不足：

```json
{
  "code": "HEAP_LOW",
  "msg": "build csi json failed",
  "state": "uploading",
  "ts": 300
}
```

排查判断：

- C 板明明异常但小程序不提示：检查是否发布 `up/fault`，topic 是否正确。
- 后端收到 fault 后没有自动 stop：检查 fault 发生时数据库是否有 `current_session`，且 `detection_state` 是否为 `starting/running/stopping`。
- 后端自动 stop 后硬件无响应：检查 C 板是否订阅 `down/control`。

## 4. 控制逻辑：用户点击开始检测

### 小程序动作

1. 用户在设备详情页点击开始检测。
2. 小程序请求：

```text
POST /api/v1/devices/{device_name}/control
payload: {"action":"start"}
```

3. 小程序先显示“正在启动”。
4. 小程序等待 WebSocket 或轮询结果。

### Python 后端动作

1. 校验用户 Token。
2. 校验设备归属。
3. 校验设备必须：

```text
enabled = true
state = online
fault_code = null
detection_state = idle
last_status_at 未超时
```

4. 生成 session。
5. 发布 control topic：

```text
topic: csi/v1/devices/{device_name}/down/control
qos: 1
retain: false
payload:
{
  "cmd": "control",
  "action": "start",
  "session": "sess-..."
}
```

6. 将数据库更新为：

```text
current_session = sess-...
detection_state = starting
network_quality = unknown
```

### 硬件应该做什么

C 板必须订阅：

```text
csi/v1/devices/{device_name}/down/control
```

收到 start 后执行：

```text
1. 校验 JSON 中 cmd=control
2. 校验 action=start
3. 保存 session
4. 清空 CSI 缓存
5. 重置 batch 编号
6. 设置 upload=true
7. 通过 UART 向 B 板发送 CSI_START\n
8. 切换状态为 uploading
9. 发布 up/ack
10. 开始按 1 秒上传 up/csi
```

ACK payload：

```text
topic: csi/v1/devices/{device_name}/up/ack
qos: 1
retain: false
payload:
{
  "cmd": "control",
  "action": "start",
  "ok": true,
  "state": "uploading",
  "err": 0,
  "msg": "",
  "ts": 240
}
```

建议 ACK 中带回同一个 session：

```json
{
  "cmd": "control",
  "action": "start",
  "session": "sess-...",
  "ok": true,
  "state": "uploading",
  "err": 0,
  "msg": "",
  "ts": 240
}
```

排查判断：

- 小程序提示“设备离线，无法启动”：后端没有收到 online/status，先查上行状态。
- 小程序提示“设备运行状态已超时”：C 板没有持续发布 `up/status`。
- Broker 有 down/control 但硬件不启动：检查 C 板订阅 topic 是否包含正确 device_name。
- C 板启动了但小程序卡“正在启动”：检查是否发布了 `up/ack` 或 `up/status state=uploading`。

## 5. 运行逻辑：CSI 正常上传

### Python 后端动作

1. 收到 start ACK 后，将检测状态改为 `running`。
2. 收到 `up/status state=uploading` 后更新运行状态。
3. 收到 `up/csi` 后解包。
4. 根据一段时间内的帧序号、丢帧、无效帧和批次间隔计算网络质量：

```text
good / fair / poor / unknown
```

5. 网络质量变化时通过 WebSocket 通知小程序。
6. 跌倒检测算法当前可先保持占位。

### 硬件应该做什么

C 板进入 uploading 后，每 1 秒发布 CSI：

```text
topic: csi/v1/devices/{device_name}/up/csi
qos: 0
retain: false
payload:
{
  "session": "sess-...",
  "batch": 1,
  "frames": 30,
  "seq0": 1,
  "seq1": 30,
  "ts0": 123456,
  "ts1": 1123456,
  "fmt": "csib64-v2-1s",
  "bytes": 7524,
  "rssi": -45,
  "invalid": 0,
  "data": "base64_encoded_binary_csi_data",
  "ts": 240
}
```

`data` 解码后必须符合：

```text
8 字节 batch_header
16 字节 frame_header + raw CSI
16 字节 frame_header + raw CSI
...
```

batch header：

```text
magic0, magic1, version, frame_count, batch_no
```

frame header：

```text
seq, timestamp_us, rssi, first_word_invalid, csi_len
```

排查判断：

- 后端提示 CSI 格式错误：检查 `fmt`、`bytes`、Base64、batch header、frame header。
- 网络质量一直 unknown：通常 CSI 包少于两个有效批次。
- 网络质量 poor：检查 seq 是否连续、batch 是否递增、invalid 是否过高、1 秒周期是否稳定。
- 小程序没有网络质量变化：确认 WebSocket 已连接，或查看设备详情页是否轮询刷新。

## 6. 异常逻辑：CSI 上传中断

### 触发条件

启动后 C 板进入 `uploading`，但 B 板长时间没有向 C 板发送 CSI，或 C 板无法正常打包上传。

### Python 后端动作

1. 收到 `up/fault`。
2. 将设备状态改为：

```text
state = error
runtime_state = fault
detection_state = stopping
fault_code = 硬件上报 code
fault_message = 硬件上报 msg
```

3. 向小程序推送 `device.fault`。
4. 自动下发：

```text
topic: csi/v1/devices/{device_name}/down/control
payload:
{
  "cmd": "control",
  "action": "stop",
  "session": "原 session"
}
```

5. 收到 stop ACK 后，将 `detection_state` 收敛为 `idle`，但保留 `state=error` 和故障信息。
6. 等设备重新上线发布 online 后，后端清除旧故障。

### 硬件应该做什么

1. 检测到 UART 超时或内部错误后，立即发布 `up/fault`。
2. 继续保持 MQTT 可用，等待后端自动下发 stop。
3. 收到 stop 后执行：

```text
1. 通过 UART 给 B 板发送 CSI_STOP\n
2. upload=false
3. 清空 CSI 缓存
4. 切换为 idle 或 fault 后等待恢复
5. 发布 up/ack action=stop
```

stop ACK：

```json
{
  "cmd": "control",
  "action": "stop",
  "session": "sess-...",
  "ok": true,
  "state": "idle",
  "err": 0,
  "msg": "",
  "ts": 260
}
```

排查判断：

- fault 后 C 板仍持续上传 CSI：检查 stop control 是否收到、UART `CSI_STOP\n` 是否发出。
- fault 后小程序没有异常提示：检查后端是否收到 fault topic，或者 WebSocket 是否断开。
- fault 解除后小程序仍显示异常：设备需要重新发布 `up/online {"status":"online"}`，后端才会清除旧故障。

## 7. 用户主动停止检测

### 小程序动作

1. 用户在设备详情页点击停止检测。
2. 小程序请求：

```text
POST /api/v1/devices/{device_name}/control
payload: {"action":"stop"}
```

3. 小程序显示“正在停止”。

### Python 后端动作

1. 校验当前 `detection_state` 必须是 `starting` 或 `running`。
2. 校验存在 `current_session`。
3. 下发：

```text
topic: csi/v1/devices/{device_name}/down/control
payload:
{
  "cmd": "control",
  "action": "stop",
  "session": "当前 session"
}
```

4. 数据库先改为：

```text
detection_state = stopping
network_quality = unknown
```

5. 以后端收到硬件 ACK 为准，最终改为 `idle`。

### 硬件应该做什么

C 板收到 stop 后：

```text
1. 校验 action=stop
2. 校验或复用 session
3. 通过 UART 向 B 板发送 CSI_STOP\n
4. upload=false
5. 清空 CSI 缓存
6. 状态切换 idle
7. 发布 up/ack action=stop
8. 停止 up/csi，继续每 5 秒发布 up/status
```

停止后的 status：

```json
{
  "state": "idle",
  "session": "",
  "uart": true,
  "upload": false,
  "rssi": -45,
  "heap": 158720,
  "uptime": 260000,
  "ts": 260
}
```

排查判断：

- 小程序卡“正在停止”：检查 `up/ack action=stop` 或 `up/status state=idle` 是否发布。
- 后端拒绝 stop：通常数据库中没有正在运行的 `current_session`。
- C 板 stop 后仍发 CSI：检查 `upload=false` 是否生效，B 板是否收到 `CSI_STOP\n`。

## 8. 推荐端到端验收顺序

### 8.1 不接硬件的后端基础验证

```text
1. 后端能启动
2. 小程序能登录
3. users 表出现微信用户
4. 授权手机号后 users.phone 写入
5. 管理员能创建设备并绑定用户
6. 小程序能看到绑定设备
```

### 8.2 接 Broker 和 C 板

```text
1. Mosquitto 启动
2. 后端 .env 中 MQTT_ENABLED=1
3. C 板连接 Wi-Fi/MQTT
4. Broker 能看到 up/online、up/wifi、up/status
5. 后端数据库 devices.state 从 offline 变 online
6. 小程序首页显示设备在线
```

### 8.3 启动检测

```text
1. 小程序点击开始
2. Broker 看到 down/control action=start
3. C 板向 B 板发送 CSI_START\n
4. Broker 看到 up/ack action=start
5. Broker 看到 up/status state=uploading
6. Broker 看到 up/csi 每秒上传
7. 后端 detection_state 变 running
8. 小程序显示检测运行中
9. 网络质量从 unknown 变 good/fair/poor
```

### 8.4 故障验证

```text
1. 启动检测后断开 B->C UART 或停止 B 板 CSI
2. C 板发布 up/fault code=UART_TIMEOUT
3. 后端 state=error，runtime_state=fault
4. 小程序显示设备异常
5. 后端自动下发 down/control action=stop
6. C 板执行 CSI_STOP 并发布 stop ACK
```

### 8.5 主动停止

```text
1. 小程序点击停止检测
2. Broker 看到 down/control action=stop，session 与 start 相同
3. C 板向 B 板发送 CSI_STOP\n
4. Broker 看到 up/ack action=stop
5. Broker 看到 up/status state=idle
6. 后端 detection_state=idle
7. 小程序显示等待启动
```

## 9. 常见问题定位表

| 现象 | 优先检查 |
|---|---|
| 小程序登录失败 | `WECHAT_APPID`、`WECHAT_SECRET`、小程序 AppID |
| 小程序没有设备 | 管理员是否创建 `devices` 并绑定 `owner_user_id` |
| 设备一直离线 | C 板是否发布 `up/online`，device_name 是否一致 |
| 设备在线但无法启动 | 后端是否收到最近 `up/status`，设备是否有 fault |
| 点击开始后硬件不动 | C 板是否订阅正确 `down/control` |
| 硬件启动但 UI 卡 starting | C 板是否发布 `up/ack start` 或 `up/status uploading` |
| CSI 上传但网络质量 unknown | 有效 CSI 批次数是否不足两个 |
| CSI 格式被后端丢弃 | `fmt`、`bytes`、Base64、batch header、frame header |
| 故障后没有自动停止 | fault 发生时是否存在 current_session |
| 停止后仍上传 CSI | C 板 `upload=false` 是否生效，B 板是否收到 `CSI_STOP\n` |
| 后端重启后状态不对 | `up/online` 和 `up/wifi` 是否 retain=true |

## 10. 硬件侧最小自检清单

每次烧录 C 板后，建议逐项确认：

- C 板 MQTT 能连接 Broker。
- C 板 Client ID 为 `esp_{device_name}_001`。
- C 板订阅 `csi/v1/devices/{device_name}/down/control`。
- C 板上线发布 `up/online`，QoS 1，retain true。
- C 板离线 LWT 发布 `up/online {"status":"offline"}`，QoS 1，retain true。
- C 板每 5 秒发布 `up/status`。
- C 板收到 start 后向 B 板发送 `CSI_START\n`。
- C 板收到 stop 后向 B 板发送 `CSI_STOP\n`。
- C 板每次 control 都发布 `up/ack`。
- C 板 uploading 后每 1 秒发布 `up/csi`。
- C 板 UART 超时或内存不足时发布 `up/fault`。
- C 板所有 payload 都是合法 JSON。
- C 板所有 `session` 与后端下发 session 保持一致。
