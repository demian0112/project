# CSI 跌倒检测项目文档

## 1. 项目概览

本项目由两个主要子项目组成：

- `csi_miniprogram--`：微信小程序前端，使用 TypeScript、WXML、LESS 开发。
- `ESP32-MQTT-Backend-main`：Flask 后端，负责小程序接口、后台管理页面、SQLite 数据库、WebSocket 实时推送，以及与 ESP32/采集设备之间的 MQTT 通信。

整体业务目标是：用户在小程序中查看家庭设备状态、启动或停止 CSI 检测；硬件通过 MQTT 上传在线状态、运行状态、CSI 数据、控制回执和故障；后端维护设备权威状态快照，检测跌倒结果并推送到小程序。

核心数据链路：

```text
微信小程序
  -> /api/v1 REST 接口
  -> Flask 后端 / SQLite
  -> DeviceCoordinator
  -> MQTT Broker
  -> ESP32/CSI 采集设备

ESP32/CSI 采集设备
  -> MQTT 上行主题
  -> Flask 后端解析与状态更新
  -> SQLite 快照 / WebSocket 推送
  -> 小程序实时刷新和告警跳转
```

### 2.2 环境与接口配置

配置位于 `miniprogram/config/env.ts`。

### 2.3 启动与登录流程

入口逻辑在 `miniprogram/app.ts`。

小程序启动时执行：

1. 清空业务缓存状态。
2. 从本地存储读取 token。
3. 如果 token 存在，设置全局登录态并拉取业务数据。
4. 如果 token 不存在或业务数据拉取失败，调用 `wx.login`。
5. 将微信 `code` 发送到后端 `/api/v1/auth/wechat-login`。
6. 后端返回 access token 后写入本地存储。
7. 并发拉取用户、设备、跌倒事件。
8. 建立 WebSocket 实时连接。

全局数据主要包括：

- `token`
- `user`
- `devices`
- `fallEvents`
- `authReady`
- `bootstrapReady`
- `bootstrapError`

认证等待逻辑在 `services/auth.ts`。接口请求除登录接口外，都会先等待 `authReady`，避免页面在登录未完成时抢先请求业务接口。

主要接口函数：

### 2.5 实时通信设计

WebSocket 客户端位于 `miniprogram/services/realtime.ts`。

连接方式：

```text
ws://127.0.0.1:5000/ws/v1/events?token=<access_token>
```

前端设计要点：

- token 通过 query 参数传给后端鉴权。
- 连接成功后通知页面重新加载数据。
- 消息统一解析为 `RealtimeEvent`。
- 连接异常时自动重连，重连间隔指数退避，最大 30 秒。
- 如果未配置 WebSocket 地址，首页和设备详情页会退化为 5 秒轮询。

实时事件类型定义在 `models/domain.ts`，包括：

- `connection.ready`
- `device.state.changed`
- `device.runtime.changed`
- `device.fault`
- `detection.network-quality`
- `detection.fall-result`

### 2.6 页面交互逻辑

#### 2.6.1 监测首页

代码位于 `pages/index/index.ts`。

首页展示：

- 全部设备数量。
- 在线设备数量。
- 检测中设备数量。
- 最近一条待处理跌倒告警。
- 设备列表、设备状态、检测状态、CSI 网络质量。

页面进入时：

1. 订阅 WebSocket 事件。
2. 加载设备列表。
3. 加载最近跌倒告警。
4. WebSocket 不可用时开启 5 秒轮询。

收到实时事件时：

- `device.fault`：如果是 CSI 采集异常，弹窗提示用户去设备详情处理。
- `detection.fall-result` 且 `fall_detected = true`：跳转到跌倒告警页。
- 其他状态事件：重新加载首页数据。

#### 2.6.2 设备详情页

代码位于 `pages/device-detail/index.ts`。

设备详情页是小程序中最重要的控制页面，负责：

- 展示设备在线/离线/异常。
- 展示检测状态。
- 展示 CSI 网络质量。
- 展示最近通信时间。
- 展示待处理跌倒告警。
- 启动检测。
- 停止检测。
- 设备故障复位。

设备控制流程：

1. 用户点击“开始检测”或“停止检测”。
2. 前端先检查设备是否在线、是否启用、是否存在故障。
3. 开始检测前，请求微信订阅消息授权，包含跌倒提醒和设备故障提醒。
4. 前端调用 `/api/v1/devices/<device_name>/control`。
5. 后端返回 202，表示命令已经发布到 MQTT。
6. 前端将页面置为 `starting` 或 `stopping`。
7. 前端启动短轮询，每 1.5 秒确认状态，最多等待 15 秒。
8. 如果 WebSocket 收到运行状态变化，也会触发重新加载。
9. 设备最终进入 `running` 或 `idle` 后，前端解除 loading。

这里有一个重要设计：接口返回成功只代表后端已经发布控制命令，不代表硬件已经执行成功。最终状态以后端收到硬件 `ack`、`status` 或 `csi` 后更新为准。

故障复位流程：

1. 页面展示设备采集异常 banner。
2. 用户点击“我已检查，恢复设备”。
3. 前端弹窗确认用户已检查供电、A/B 板连接和设备摆放。
4. 调用 `/api/v1/devices/<device_name>/reset-fault`。
5. 后端通过 MQTT 发布 `reset`。
6. 硬件 ACK 成功或上报 idle 后，后端清除故障。
7. WebSocket 推送 `device.runtime.changed`，前端提示设备已恢复。

#### 2.6.3 跌倒告警页

代码位于 `pages/fall-alert/index.ts`。

页面通过 query 中的 `id` 加载跌倒事件。由于前端当前没有单条告警接口，页面会拉取最多 100 条事件，再在前端按 id 查找。

用户可以：

- 查看设备、位置、发生时间、网络质量。
- 点击“我已确认安全”，将事件状态更新为 `confirmed`。
- 返回首页。

#### 2.6.4 我的页面

代码位于 `pages/mine/mine.ts`。

主要能力：

- 展示用户资料。
- 如果未登录，显示登录入口。
- 绑定微信手机号。
- 查看设备数量。
- 进入设备管理页。
- 请求跌倒提醒订阅消息授权。
- 拉取订阅状态并展示。

手机号绑定使用微信 `getPhoneNumber` 返回的一次性 `code`，发送到后端 `/api/v1/me/phone`，由后端调用微信接口换取手机号。

#### 2.6.5 设备管理页

代码位于 `pages/device-management/index.ts`。

普通用户可维护自己设备的：

- 显示名称。

## 4. WebSocket 实时事件设计

WebSocket 路由位于 `services/websocket_service.py`：

```text
/ws/v1/events?token=<access_token>
```

后端建立连接时：

1. 解析 token。
2. 校验用户存在且状态为 active。
3. 将 websocket 加入 `WebSocketHub` 中按用户分组的连接池。
4. 发送 `connection.ready`。
5. 循环等待客户端消息，连接关闭后移除。

推送消息结构：

```json
{
  "event": "device.runtime.changed",
  "device_name": "device001",
  "data": {}
}
```

事件按用户推送，只有设备所属用户会收到相关设备事件。

常见事件：

- `device.state.changed`：设备 online/offline/error 主状态变化。
- `device.runtime.changed`：硬件运行状态或检测状态变化。
- `device.fault`：设备进入故障。
- `detection.network-quality`：CSI 网络质量变化。
- `detection.fall-result`：算法识别到跌倒。

## 5. MQTT 交互逻辑

### 5.1 MQTT 连接设计

MQTT 配置位于 `app/mqtt/config.py`。

默认配置：

```text
host     = 192.168.101.48
port     = 1883
username = csi_user
password = ""
keepalive = 60
```

可通过环境变量覆盖：

- `MQTT_BROKER_HOST` 或 `MQTT_HOST`
- `MQTT_BROKER_PORT` 或 `MQTT_PORT`
- `MQTT_BROKER_USERNAME` 或 `MQTT_USERNAME`
- `MQTT_BROKER_PASSWORD` 或 `MQTT_PASSWORD`
- `MQTT_KEEPALIVE`

后端使用 Paho MQTT。设计上每个数据库设备拥有一个独立的 `DeviceMqttClient`，client id 形如：

```text
python_<device_name>_001
```

`MqttManager` 管理所有设备客户端：

- `ensure_device(device_name)`：确保某设备 MQTT 客户端已创建并异步连接。
- `remove_device(device_name)`：移除并断开设备客户端。
- `publish_control(...)`：向设备下行控制主题发布控制命令。

如果 `MQTT_ENABLED` 为 false，后端不会真实连接 MQTT，控制设备时会返回 `MQTT_UNAVAILABLE`。

### 5.2 MQTT Topic 规范

topic 生成逻辑位于 `app/mqtt/topics.py`。

设备名规则：

```text
^[A-Za-z0-9_-]{1,32}$
```

统一前缀：

```text
csi/v1/devices
```

上行 topic：

```text
csi/v1/devices/{device_name}/up/online
csi/v1/devices/{device_name}/up/wifi
csi/v1/devices/{device_name}/up/status
csi/v1/devices/{device_name}/up/csi
csi/v1/devices/{device_name}/up/ack
csi/v1/devices/{device_name}/up/fault
```

下行控制 topic：

```text
csi/v1/devices/{device_name}/down/control
```

后端订阅采用精确 topic，不使用通配符。

QoS 设计：

| Topic 名称 | 方向 | QoS | 说明 |
| --- | --- | --- | --- |
| `online` | 上行 | 1 | 设备在线/离线 |
| `wifi` | 上行 | 1 | Wi-Fi 心跳或连通性 |
| `status` | 上行 | 1 | 硬件运行状态 |
| `csi` | 上行 | 0 | 高频 CSI 数据 |
| `ack` | 上行 | 1 | 控制命令回执 |
| `fault` | 上行 | 1 | 设备故障 |
| `control` | 下行 | 1 | start/stop/reset 控制 |

### 5.3 下行控制 payload

控制命令由 `DeviceMqttClient.publish_control()` 发布到：

```text
csi/v1/devices/{device_name}/down/control
```

通用 JSON：

```json
{
  "cmd": "control",
  "action": "start",
  "session": "sess-20260709-120000-abcdef"
}
```

支持 action：

- `start`
- `stop`
- `reset`
- `clear_fault`

其中：

- `start` 如果没有 session，后端会自动生成 session。
- `stop` 必须带当前 session。
- `reset` 和 `clear_fault` 可额外带 `command_id`、`reason`、`source`。

复位命令示例：

```json
{
  "cmd": "control",
  "action": "reset",
  "session": "sess-xxx",
  "command_id": "cmd-xxx",
  "reason": "user_fault_confirm",
  "source": "user"
}
```

### 5.4 上行 online 处理

处理函数：`DeviceCoordinator._handle_online()`。

设备可上报：

```json
{
  "status": "online"
}
```

也兼容：

```json
{
  "online": true
}
```

或：

```json
{
  "state": "offline"
}
```

后端会将字符串、数字和布尔值统一解析为布尔在线状态。

在线时：

- 更新 `last_seen_at`。
- 更新 `last_online_at`。
- 如果设备不在故障中，将 `state` 置为 `online`。
- 推送 `device.state.changed`。

离线时：

- 如果设备正在检测，并且距离最后一次有效 CSI 或状态信号未超过硬超时，后端会忽略瞬时离线，避免硬件短暂抖动导致误判。
- 否则调用 `_mark_device_offline()`，清空检测 session、网络质量和故障状态。
- 推送 `device.state.changed`。

### 5.5 上行 wifi 处理

处理函数：`DeviceCoordinator._handle_wifi()`。

有效 payload：

```json
{
  "ok": true
}
```

如果 `ok` 为 true：

- 更新 `last_seen_at`。
- 如果设备不在故障中，置为 `online`。
- 更新 `last_online_at`。

### 5.6 上行 status 处理

处理函数：`DeviceCoordinator._handle_status()`。

硬件状态只接受：

- `booting`
- `idle`
- `uploading`
- `fault`

示例：

```json
{
  "state": "uploading",
  "session": "sess-xxx"
}
```

状态映射逻辑：

- `uploading`：表示硬件正在上传 CSI。后端会将检测状态推进到 `running`，并绑定 session。
- `booting`：表示硬件重启中。后端会将检测状态重置为 `idle`，清空 session 和网络质量。
- `idle`：如果当前正在 starting/running/stopping，说明硬件已停止检测，后端会重置为 `idle` 并清空 session。
- `fault`：进入故障流程，记录故障码和故障消息。

如果设备已经处于故障状态，普通的非故障 status 不会覆盖故障，只更新时间戳。这可以避免硬件后续状态包把故障误清掉。

复位特殊逻辑：

- 用户发出 reset 后，后端记录 `_fault_reset_pending`。
- 如果之后收到 `state=idle` 且匹配 pending reset，后端会清除故障。
- 清除后推送 `device.runtime.changed`，并带上 `fault_cleared = true`。

### 5.7 上行 ack 处理

处理函数：`DeviceCoordinator._handle_ack()`。

控制回执示例：

```json
{
  "cmd": "control",
  "action": "start",
  "ok": true,
  "session": "sess-xxx",
  "state": "uploading",
  "err": 0,
  "msg": ""
}
```

后端会校验：

- `cmd` 必须为 `control`。
- `action` 必须是 `start`、`stop`、`reset` 或 `clear_fault`。
- `ok` 必须能解析成布尔。
- 如果 ACK 带 session，必须与当前 session 一致，否则视为过期 ACK 并忽略。

`start` ACK：

- 成功：`runtime_state` 设为 `uploading` 或 ACK 中合法状态，`detection_state` 设为 `running`。
- 失败：检测状态回到 `idle`，清空 session 和网络质量。

`stop` ACK：

- 成功：检测状态回到 `idle`，清空 session 和网络质量。
- 失败：检测状态回到 `running`。

`reset` 或 `clear_fault` ACK：

- 成功：调用 `_clear_fault()`，清除故障，设备回到 online + idle。
- 失败：保持 error/fault 状态。

ACK 处理完成后会推送 `device.runtime.changed`。

### 5.8 上行 fault 处理

处理函数：`DeviceCoordinator._handle_fault()` 和 `_record_fault()`。

故障上报示例：

```json
{
  "session": "sess-xxx",
  "code": "NO_CSI_FRAME_TIMEOUT",
  "msg": "启动检测后未收到 CSI 数据"
}
```

后端处理：

1. 校验 session，过期 session 的故障会被忽略。
2. 设置设备：
   - `state = error`
   - `runtime_state = fault`
   - `fault_code = code`
   - `fault_message = message`
3. 如果设备正在检测，将 `detection_state` 置为 `stopping`。
4. 清理当前 session 的 CSI 质量窗口和算法窗口。
5. 限流后推送 `device.fault`。
6. 对特定故障码尝试发送微信设备故障通知。
7. 如果当前存在检测 session，后端会自动发布一次 `stop` 控制命令，尝试让硬件停止上传。

会触发微信设备故障通知的故障码：

- `NO_CSI_FRAME`
- `NO_CSI_FRAME_TIMEOUT`
- `UART_TIMEOUT`

前端也会对这些故障码弹出“设备采集异常”提示。

### 5.9 上行 csi 处理

处理函数：`DeviceCoordinator._handle_csi()`。

CSI payload 解码位于 `services/csi_payload_service.py`。

当前支持格式：

```text
csib64-v2
```

JSON 外层字段包括：

```json
{
  "session": "sess-xxx",
  "fmt": "csib64-v2",
  "batch": 1,
  "frames": 45,
  "seq0": 1,
  "seq1": 45,
  "ts0": 1000000,
  "ts1": 2500000,
  "bytes": 12345,
  "data": "base64..."
}
```

`data` 是 Base64 编码后的二进制 CSIB 批次。二进制格式中包含：

- 批次头：magic、版本、帧数量、采样率、batch_no、起止时间戳。
- 每帧头：sequence、timestamp_us、rssi、first_word_invalid、CSI 长度。
- 每帧原始 CSI bytes。

后端会严格校验：

- magic 必须是 `CSIB`。
- 二进制版本必须匹配。
- JSON 中的帧数、batch、seq、ts、bytes 必须与二进制内容一致。
- 帧序号必须递增。
- 时间戳不能倒退。
- 单批最大帧数 80。
- 二进制最大 512 KB。
- 单帧 CSI 最大 384 bytes。

成功解码后：

1. 更新 `last_seen_at` 和 `last_csi_at`。
2. 设置 `runtime_state = uploading`。
3. 设置 `detection_state = running`。
4. 设置 `state = online`。
5. 交给 `CsiQualityTracker` 评估网络质量。
6. 将批次转为算法输入，放入内存滑动窗口。
7. 窗口帧数达到 `CSI_WINDOW_SIZE` 后调用跌倒算法。
8. 如果算法返回 1，创建 `FallEvent`。
9. 推送 `detection.fall-result`。

当前 `services/fall_detect_service.py` 中的默认算法是安全占位实现：

```text
predict_fall(...) -> 0
```

也就是说，默认不会产生跌倒事件。项目通过 `app.config["FALL_PREDICTOR"]` 预留了算法替换入口，测试中会注入返回 1 的 predictor 来验证事件生成流程。

### 5.10 CSI 网络质量评估

质量评估位于 `services/csi_quality_service.py`。

后端按设备和 session 维护滚动窗口，使用这些指标判断质量：

- 批次间隔。
- 帧序号连续性。
- 丢帧率。
- invalid 帧比例。
- 是否出现非单调 batch 或 seq。
- 最大间隔是否超过软超时。

质量等级：

- `good`：丢帧和 invalid 很低，间隔稳定。
- `fair`：有轻微丢帧或间隔拉长，但仍可接受。
- `poor`：质量较差、间隔异常或数据问题明显。
- `unknown`：数据不足。

如果硬件重启导致 seq 回到很小值，后端会识别为 `seq_reset`，清空算法窗口，并将质量临时标为 `fair`，避免新旧数据混在一起。

### 5.11 离线与 CSI 超时监控

`DeviceCoordinator.start_monitor()` 会启动后台线程，每 2 秒扫描一次设备。

普通离线：

- 如果设备不在检测中，且 `last_seen_at` 超过 `OFFLINE_TIMEOUT_SECONDS`，标记为离线。

检测中 CSI 超时：

- 如果处于 starting/running/uploading，优先看 `last_csi_at`。
- 在启动宽限期内，不立刻报错。
- 超过软超时，降低网络质量。
- 超过恢复宽限期，继续保持 running，但质量为 poor。
- 超过硬超时，如果设备仍应处于检测状态，记录 `NO_CSI_FRAME_TIMEOUT` 故障。

这套设计避免因为短暂网络抖动把正在运行的设备直接踢成离线，同时又能在长时间无 CSI 时进入明确故障状态。

## 6. 核心业务闭环

### 6.1 用户登录闭环

```text
小程序 wx.login
  -> /api/v1/auth/wechat-login
  -> 后端 code2Session 换 openid
  -> 创建或更新 User
  -> 签发 access_token
  -> 小程序保存 token
  -> 拉取用户、设备、跌倒事件
  -> 建立 WebSocket
```

### 6.2 启动检测闭环

```text
用户点击开始检测
  -> 小程序请求订阅消息授权
  -> POST /api/v1/devices/{device}/control action=start
  -> DeviceCoordinator 校验设备在线、启用、无故障、空闲
  -> 生成 session 和 command_id
  -> MQTT 发布 down/control start
  -> 数据库标记 detection_state=starting
  -> 小程序显示正在启动
  -> 设备 ACK start ok 或 status uploading 或 CSI 到达
  -> 后端标记 running
  -> WebSocket 推送 device.runtime.changed
  -> 小程序刷新为检测运行中
```

### 6.3 停止检测闭环

```text
用户点击停止检测
  -> POST /api/v1/devices/{device}/control action=stop
  -> 后端校验当前 starting/running 且有 session
  -> MQTT 发布 down/control stop
  -> 数据库标记 detection_state=stopping
  -> 设备 ACK stop ok 或 status idle
  -> 后端清空 session，状态回到 idle
  -> WebSocket 推送
  -> 小程序刷新
```

### 6.4 跌倒事件闭环

```text
设备上传 CSI
  -> 后端解码 csib64-v2
  -> 更新设备 last_csi_at 和网络质量
  -> CSI 批次进入内存算法窗口
  -> predictor 返回 1
  -> 创建 FallEvent(status=pending)
  -> WebSocket 推送 detection.fall-result
  -> 小程序跳转跌倒告警页
  -> 用户确认安全
  -> PATCH /api/v1/fall-events/{id}
  -> 后端标记 confirmed，写 handled_at
```

注意：代码中微信跌倒通知服务 `send_fall_alert()` 已实现，并且管理端模拟跌倒会调用它；CSI 自动识别跌倒路径当前会落库并推 WebSocket，但在 `device_state_service.py` 中没有直接调用 `send_fall_alert()`。

### 6.5 故障与复位闭环

```text
设备上报 fault 或后端检测 CSI 硬超时
  -> 后端记录 fault_code/fault_message
  -> 设备 state=error runtime_state=fault
  -> 如果有运行 session，自动 MQTT 发布 stop
  -> WebSocket 推送 device.fault
  -> 对指定 CSI 故障码尝试微信设备故障通知
  -> 小程序弹出采集异常
  -> 用户检查硬件后点击恢复设备
  -> POST /api/v1/devices/{device}/reset-fault
  -> MQTT 发布 reset
  -> 设备 ACK reset ok 或 status idle
  -> 后端清除故障并恢复 online/idle
  -> 小程序刷新
```

## 7. 微信订阅消息设计

前端在两个地方请求订阅消息：

- 设备详情页开始检测前，同时请求跌倒提醒和设备故障提醒。
- 我的页面中，用户可单独开启跌倒提醒。

授权结果会通过 `/api/v1/wechat/subscriptions` 保存到 `WxSubscription`。

后端发送前会检查：

- `WECHAT_NOTIFY_ENABLED` 是否开启。
- 对应模板 ID 是否配置。
- 用户是否有 openid。
- 用户是否授权对应模板。
- 授权状态是否为 `accept`。

设备故障通知会在后端记录故障时自动尝试发送。

跌倒提醒通知服务已经实现，管理端模拟跌倒路径会调用。自动 CSI 跌倒识别路径当前主要依靠 WebSocket 实时推送。

## 8. 数据一致性与容错设计

### 8.1 设备状态以数据库为准

前端展示的设备状态来自后端数据库快照，而不是直接依赖 MQTT 消息。MQTT 只负责驱动后端更新快照。

这样做的好处：

- 小程序刷新后仍能看到最近状态。
- WebSocket 丢消息后可以重新拉取 REST 接口恢复状态。
- 后台管理端和小程序看到的是同一套数据。

### 8.2 控制请求不是最终状态

控制接口返回 202 只表示后端已接收并尝试发布 MQTT 命令。硬件最终是否执行成功，取决于：

- `ack` 回执。
- `status` 上报。
- `csi` 是否开始到达。
- 后端超时扫描结果。

所以前端采用“乐观 pending 状态 + WebSocket + 短轮询确认”的组合。

### 8.3 Session 防串扰

检测任务通过 `current_session` 绑定。

后端会忽略：

- 过期 session 的 CSI。
- 过期 session 的 ACK。
- 过期 session 的 fault。
- 上传中报告了不一致 session 的 status。

这可以避免硬件重启、旧包延迟或重复发布影响当前任务。

### 8.4 原始 CSI 不落库

CSI 原始数据只在内存中短暂存在，用于质量评估和算法窗口。数据库只保存：

- 设备最新状态。
- 网络质量等级。
- 跌倒事件结果。
- 通知日志。

这减少了数据库压力，也避免 SQLite 承担高频大体量 CSI 存储。

## 10. 后续可优化点

结合代码现状，后续可以考虑：

- 增加单条跌倒事件详情接口，避免告警页拉取 100 条后前端查找。
- 将 CSI 自动跌倒识别路径接入 `send_fall_alert()`，如果业务希望跌倒也走微信服务通知。
- 将生产环境 `apiBaseUrl` 和 `wsBaseUrl` 补齐，避免发布时为空。
- 引入真实跌倒算法替换 `fall_detect_service.predict_fall()` 默认占位实现。
- 如果后续要长期保存 CSI 原始数据，建议独立使用时序存储或文件对象存储，不要直接塞入当前 SQLite 主库。
