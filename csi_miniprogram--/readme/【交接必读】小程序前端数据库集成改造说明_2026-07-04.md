## 1. 本次改造目标

本次改造将小程序从本地演示模式调整为真实后端和数据库驱动模式。

核心原则：

1. 微信用户启动小程序后直接进行微信登录，不提供独立注册功能。
2. 小程序不直接访问数据库，由 Flask API 查询和更新数据库。
3. 后端根据微信 `openid` 查询用户；用户不存在时自动新增，已存在时更新登录时间。
4. 用户、设备状态和跌倒事件全部以后端数据库返回为准。
5. 前端不生成虚拟用户、虚拟设备、模拟状态或模拟告警。

---

## 2. 冷启动流程

当前冷启动流程如下：

```text
小程序启动
→ 清空内存中的旧业务数据
→ wx.login() 获取一次性 code
→ POST /api/v1/auth/wechat-login
→ 后端调用微信 code2Session
→ 后端查询 users
→ 用户不存在则新增，存在则更新 last_login_at
→ 前端保存 access_token
→ 并行请求 /me、/devices、/fall-events
→ 建立 WebSocket
→ 页面展示数据库真实数据
```

首次用户即使没有昵称或头像，后端也已经完成用户身份建档。前端显示“未设置昵称”等空状态，不使用虚拟昵称冒充真实资料。

主要实现文件：

- `miniprogram/app.ts`
- `miniprogram/services/auth.ts`
- `miniprogram/utils/api.ts`

## 3. 首屏读取的数据

登录成功后立即请求：

```http
GET /api/v1/me
GET /api/v1/devices
GET /api/v1/fall-events?limit=20
```

对应数据库来源：

| 页面数据 | 后端来源 |
|---|---|
| 用户资料 | `users` |
| 用户绑定设备 | `devices.owner_user_id` |
| 设备在线和故障状态 | `devices.state` |
| 检测状态 | `devices.detection_state` |
| CSI 网络质量 | `devices.network_quality` |
| 跌倒事件 | `fall_events` |

数据库没有设备时，首页显示“暂无设备”；请求失败时显示错误与重试按钮，不加载本地演示设备。

## 4. 已删除的演示逻辑

已删除以下内容：

- legacy 旧版 FastAPI `/api/device/*` 兼容分支。
- 默认用户、固定昵称和固定头像。
- 固定客厅设备和固定 ESP 设备 ID。
- 模拟在线、离线和运行状态。
- 模拟 CSI 网络质量。
- 模拟跌倒事件、置信度和持续时间。
- API 失败后回退到 Mock 数据的逻辑。
- 本地缓存跌倒事件处理结果的逻辑。
- 微信模板项目遗留的 logs 页面。

允许保留的内容只有 UI 空状态文案，例如“暂无设备”“未设置位置”“暂无数据”。这些文案不代表业务数据。

## 5. 页面职责

### 5.1 首页 `pages/index/index`

- 展示数据库返回的设备列表。
- 展示设备在线、离线、异常状态。
- 展示检测状态和 CSI 网络质量。
- 展示待处理跌倒告警入口。
- 点击设备进入设备监测详情。
- 首页不直接执行开始或停止检测。

### 5.2 设备监测详情 `pages/device-detail/index`

- 展示实时检测仪表。
- 展示设备权威状态、故障和最近通信时间。
- 展示 CSI 网络质量。
- 执行开始和停止检测。
- 展示当前设备的待处理跌倒告警。

### 5.3 设备管理 `pages/device-management/index`

- 只展示设备名称、设备编号和安装位置。
- 不展示在线状态、检测状态和网络质量。
- 不提供开始或停止检测。
- 设备由管理员网页创建并绑定用户，小程序没有绑定或解绑接口。

### 5.4 个人中心 `pages/mine/mine`

- 展示 `/me` 返回的真实用户资料。
- 昵称或头像为空时显示明确空状态。
- 只保留设备管理入口。

### 5.5 跌倒告警 `pages/fall-alert/index`

- 从 `/fall-events` 查询真实事件。
- 根据事件 ID 查找对应记录。
- 使用 `PATCH /api/v1/fall-events/{id}` 更新处理状态。

## 6. 当前 API 契约

```text
POST  /api/v1/auth/wechat-login
GET   /api/v1/me
PATCH /api/v1/me/profile
GET   /api/v1/devices
GET   /api/v1/devices/{device_name}
POST  /api/v1/devices/{device_name}/control
GET   /api/v1/fall-events
PATCH /api/v1/fall-events/{id}
WS    /ws/v1/events?token=<access_token>
```

除微信登录外，其余接口统一携带：

```http
Authorization: Bearer <access_token>
```

接口响应字段以 `微信小程序前端与后端API及数据库集成说明_最新版.md` 为准，不要根据旧交接文档恢复 legacy 接口。

## 7. 设备控制状态机

每次控制请求生成新的 `Idempotency-Key`。

```text
点击开始
→ POST control { action: start }
→ 后端返回 202 accepted
→ 前端显示 starting 并禁用按钮
→ 等待 WebSocket 或重新查询设备详情
→ detection.state = running 后显示检测中
```

停止流程同理，状态为 `stopping → idle`。

注意：

- `accepted = true` 只代表命令已发布，不代表硬件已经执行。
- 等待最终状态期间禁止重复点击。
- 15 秒仍未获得最终状态时，提示状态确认超时。
- 离线、停用或存在故障的设备不能控制。

## 8. WebSocket

当前支持：

```text
connection.ready
device.state.changed
device.runtime.changed
device.fault
detection.network-quality
detection.fall-result
```

WebSocket 只作为变化通知，不作为永久数据源。以下情况会重新请求 HTTP 快照：

- WebSocket 首次连接成功。
- WebSocket 断线重连成功。
- 小程序回到前台。
- 收到设备状态或跌倒事件。

实现文件：`miniprogram/services/realtime.ts`。

## 9. 联调前必须配置

后端地址目前故意保持为空，避免提交个人电脑 IP。

修改：

```text
miniprogram/config/env.ts
```

本地或局域网联调示例：

```typescript
development: {
  apiBaseUrl: 'http://192.168.1.20:5000',
  wsBaseUrl: 'ws://192.168.1.20:5000/ws/v1/events',
  requestTimeout: 10000,
}
```

正式发布时使用：

```text
https://正式后端域名
wss://正式后端域名/ws/v1/events
```

并在微信公众平台配置 request 和 socket 合法域名。

## 10. 当前限制和注意事项

1. 当前没有小程序侧设备绑定、解绑、重命名或修改位置接口。
2. 设备必须由管理员网页创建，并通过 `owner_user_id` 绑定用户。
3. 新用户昵称和头像可能为空，这是数据库真实状态，不应补虚拟资料。
4. `/me/profile` 已在 API 层预留，但头像上传方式需要和后端进一步确认。
5. 当前工作区的最新版 Flask 数据库后端位于同级
   `../ESP32-MQTT-Backend-main`；小程序目录内的 `backend/` 仅为历史演示，
   不参与当前联调。
6. `env.ts` 未填写地址时会显示“尚未配置后端服务地址”，这是预期行为。
7. WebSocket 不可用时，首页和设备详情会通过轮询刷新真实 HTTP 数据。
